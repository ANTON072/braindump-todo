Lesson 10: E2E テスト（Playwright）

Playwright / webServer / 代表フロー 1 本 ｜ 目安 Day 29–32（約 4 時間）

ゴール

実ブラウザで dev サーバを叩き、ログイン → タグ付き todo 作成 → 完了切替 → 削除 → ログアウトの代表フロー 1 本と、 未認証で /todos に触れるとログインへリダイレクトされることが自動で検証される状態。

学びどころ：E2E はダイヤモンドの「薄い頂点」

E2E（Playwright）← 今回
統合テスト（Lesson 5・6 で厚く積んだ）
ユニット（normalizeTagNames 等）

E2E は遅く壊れやすいので、代表フロー 1 本に絞る

認可や upsert の細かな分岐は Lesson 5・6 の統合テストで既に守られています。E2E はそれらを繰り返さず、 「画面・Server Action・DB・認証がつながって 1 本の動線として通るか」だけを代表フローで確かめます。 E2E はブラウザを起動するぶん遅く壊れやすいので、本数を増やさないのがダイヤモンドの思想です。

学びどころ：ここで初めて async Server Component の描画経路を担保する

Lesson 4 で見たとおり、app/todos/page.tsx のような async Server Component は Vitest では扱えず、公式も E2E を推奨していました。 その「ユニットで触れなかった描画経路」——requireUserId() でセッションを解決し、db.query で自分の todo を引き、 サーバーで HTML を組み立ててブラウザに返すまで——を、実ブラウザで初めて通しで検証するのがこのレッスンの役割です。

Day 29: Playwright の導入とテスト用 DB への接続

1. インストール

pnpm create playwright

対話で TypeScript・e2e ディレクトリ・ブラウザのインストールを選びます。生成される playwright.config.ts を最小構成に整えます。

2. playwright.config.ts

ポイントは 2 つ——webServer で dev サーバを自動起動し、その dev サーバをテスト用 DB に向けることです （db/index.ts は Lesson 4 で TEST_DATABASE_URL を優先する形にしてあります）：

import { config } from "dotenv";
import { defineConfig } from "@playwright/test";

config({ path: ".env.test" });

export default defineConfig({
  testDir: "./e2e",
  globalSetup: "./e2e/global-setup.ts",
  use: { baseURL: "http://localhost:3000" },
  webServer: {
    command: "pnpm dev",
    url: "http://localhost:3000",
    // CI では毎回新規に起動、ローカルでは起動済みサーバを使い回す
    reuseExistingServer: !process.env.CI,
    env: {
      // dev サーバをテスト DB に向ける（開発 DB を汚さない）
      TEST_DATABASE_URL: process.env.TEST_DATABASE_URL!,
    },
  },
});

3. globalSetup でテスト DB を掃除し、ログインできるユーザーを 1 人 seed する

E2E も統合テストと同じく、走らせる前にテスト DB を truncate して既知の状態から始めます。 このアプリは招待制で登録 UI が無い（Lesson 3）ので、「登録」は UI 操作ではなく seed で用意します。e2e/global-setup.ts：

import { sql } from "drizzle-orm";
import { db } from "@/db";
import { auth } from "@/lib/auth";

export const TEST_USER = {
  name: "E2E",
  email: "e2e@example.com",
  password: "e2e-password-123",
};

export default async function globalSetup() {
  if (!process.env.TEST_DATABASE_URL) {
    throw new Error("TEST_DATABASE_URL が未設定です。E2E はテスト用 DB でのみ実行してください。");
  }

  await db.execute(
    sql`TRUNCATE TABLE "user", todos, tags, todos_to_tags RESTART IDENTITY CASCADE`,
  );

  // 招待制のため UI 登録は無い。Lesson 3 の seed-users.ts と同じ経路でユーザーを作る
  await auth.api.signUpEmail({ body: TEST_USER });
}

disableSignup との両立

Lesson 3 で disableSignup: true にすると auth.api.signUpEmail も 403 で弾かれます（seed-users.ts と同じ制約）。 E2E で seed するには、disableSignup を環境変数で切り替え可能にして、テスト環境（.env.test）でだけ登録を許可するのが素直です。 「本番は招待制、テストは seed 可能」を env で分けます。

Day 30–31: 代表フロー 1 本を書く

4. ハッピーパス

e2e/happy-path.spec.ts。セレクタは Lesson 3・5 で書いた placeholder / ボタンラベルに合わせています：

import { expect, test } from "@playwright/test";
import { TEST_USER } from "./global-setup";

test("ログイン→タグ付き作成→完了切替→削除→ログアウト", async ({ page }) => {
  // ① ログイン
  await page.goto("/login");
  await page.getByPlaceholder("メールアドレス").fill(TEST_USER.email);
  await page.getByPlaceholder("パスワード").fill(TEST_USER.password);
  await page.getByRole("button", { name: "ログイン" }).click();
  await page.waitForURL("/todos");

  // ② タグ付きで作成
  await page.getByPlaceholder("やること").fill("歯医者に電話");
  await page.getByPlaceholder(/タグ/).fill("健康");
  await page.getByRole("button", { name: "追加" }).click();

  await expect(page.getByText("歯医者に電話")).toBeVisible();
  await expect(page.getByText("健康")).toBeVisible(); // タグ Badge も描画される

  // ③ 完了切替
  await page.getByRole("checkbox").first().check();

  // ④ 削除
  await page.getByRole("button", { name: "削除" }).first().click();
  await expect(page.getByText("歯医者に電話")).toHaveCount(0);

  // ⑤ ログアウト
  await page.getByRole("button", { name: "ログアウト" }).click();
  await page.waitForURL("/login");
});

5. 未認証リダイレクト

各テストは既定でまっさらなブラウザコンテキスト（Cookie なし）から始まります。ログインしていない状態で保護ページに触れると requireUserId() がリダイレクトすることを確かめます：

test("未認証で /todos に触れるとログインへ飛ばされる", async ({ page }) => {
  await page.goto("/todos");
  await expect(page).toHaveURL(/\/login/);
});

6. 実行

pnpm db:push:test      # テスト DB のスキーマを最新に（Lesson 4）
pnpm exec playwright test

初回はブラウザのダウンロードが走ります。--ui を付けるとタイムトラベル付きの UI モードで、どこで落ちたかを画面で追えます。

チェックポイント

  • playwright test が 2 本とも緑になる
  • dev サーバがテスト DB（5433）に向いており、開発 DB（5432）のデータが E2E で消えない
  • ハッピーパスの各ステップを --ui で 1 つずつ再生して、実ブラウザ上の描画を目で確認した
  • Lesson 5・6 の統合テストと合わせ、ダイヤモンドの 3 層（薄ユニット・太い統合・薄 E2E）が揃った

つまずきポイント

  • E2E もテスト DB を truncate してから走る。開発 DB を向いたまま実行すると確認用データが消える。globalSetup のガード（TEST_DATABASE_URL チェック）を外さない。
  • 認証状態の使い回しに注意。ログインを毎テストで繰り返すのが遅くなってきたら storageState（保存したログイン Cookie を再利用）で高速化できるが、その場合は「未認証リダイレクト」テストだけ空の context にするなど、状態の分離を意識する。まずは使い回さない素朴な形で始めてよい。
  • CI では webServer.reuseExistingServer: false。ローカルは起動済み dev を使い回して速くしたいが、CI では毎回クリーンに起動する（!process.env.CI で分岐済み）。
  • 要素が見つからず落ちるときは、getByRole / getByPlaceholder が実際のマークアップと一致しているか。playwright test --ui や --debug でセレクタを確かめる。
  • Aurora ではなくローカル dev で走らせること。E2E は外部 API（OpenAI）に触れないコア動線に限定し、ブレインダンプ抽出フローは対象外にしている（コストと不安定さを持ち込まない）。
