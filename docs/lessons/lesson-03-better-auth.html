学びどころ：Better Auth の全体像（先にこれを頭に入れる）

Better Auth の部品は 4 つで、それぞれ役割が違います。

  • サーバー設定（auth.ts）— 認証ロジック本体。DB アダプタと有効化する機能をここで宣言。
  • API ルート（/api/auth/[...all]）— サインイン等のエンドポイントを 1 つのキャッチオールで全部引き受ける。
  • クライアント（auth-client.ts）— ブラウザから API ルートを叩くための薄いラッパー。
  • DB テーブル — user / session / account / verification の 4 テーブルを Better Auth が要求し、CLI がスキーマを生成してくれる。

Day 5: サーバー設定とテーブル生成

1. インストールと環境変数

npm install better-auth

.env に追記：

BETTER_AUTH_SECRET=（openssl rand -base64 32 で生成した値）
BETTER_AUTH_URL=http://localhost:3000

2. サーバー設定

src/lib/auth.ts：

import { betterAuth } from "better-auth";
import { drizzleAdapter } from "better-auth/adapters/drizzle";
import { db } from "@/db";

export const auth = betterAuth({
  database: drizzleAdapter(db, { provider: "pg" }),
  emailAndPassword: { enabled: true },
});

3. 認証テーブルのスキーマを CLI で生成

npx @better-auth/cli@latest generate

生成された auth-schema.ts の中身（user / session / account / verification）を src/db/schema.ts に統合します（別ファイルのまま schema.ts から re-export でも可）。統合したら、Lesson 2 で保留していた FK をここで張ります：

// todos.userId と tags.userId を以下に変更
userId: text("user_id").notNull()
  .references(() => user.id, { onDelete: "cascade" }),
npx drizzle-kit generate
npx drizzle-kit migrate

学びどころ：「認証ライブラリが自前テーブルを持つ」構成

Better Auth はユーザー管理を丸ごと引き受ける代わりに、テーブル定義の主導権も持ちます。 自分のドメインテーブル（todos, tags）から user.id へ FK を張ることで両者が接続される——この「境界線」を意識するのが今回の学びです。 Better Auth のテーブルを手で改造しない（プラグインや additionalFields で拡張する）のが原則。

6. サインアップ／ログインページ

src/app/signup/page.tsx（Client Component）。ここは認証フローの学習が目的なので、Conform は使わず素朴に書きます（Conform は Lesson 4 の主役）。

"use client";

import { useState } from "react";
import { useRouter } from "next/navigation";
import { authClient } from "@/lib/auth-client";
import { Button } from "@/components/ui/button";
import { Input } from "@/components/ui/input";

export default function SignupPage() {
  const router = useRouter();
  const [errorMessage, setErrorMessage] = useState<string | null>(null);

  async function submitSignup(formData: FormData) {
    const { error } = await authClient.signUp.email({
      name: String(formData.get("name")),
      email: String(formData.get("email")),
      password: String(formData.get("password")),
    });
    if (error) setErrorMessage(error.message ?? "登録に失敗しました");
    else router.push("/todos");
  }

  return (
    <form action={submitSignup} className="mx-auto max-w-sm space-y-4 p-8">
      <Input name="name" placeholder="名前" required />
      <Input name="email" type="email" placeholder="メールアドレス" required />
      <Input name="password" type="password" placeholder="パスワード（8文字以上）" required />
      {errorMessage && <p className="text-sm text-red-600">{errorMessage}</p>}
      <Button type="submit" className="w-full">登録</Button>
    </form>
  );
}

ログインページ（/login）も同様に authClient.signIn.email で作ります。ログアウトは authClient.signOut()。

Day 7: サーバー側でのセッション取得と保護

7. セッション取得ヘルパー

以後すべての Server Action・ページで使う、このプロジェクトで最重要のヘルパーです。src/lib/session.ts：

import { headers } from "next/headers";
import { redirect } from "next/navigation";
import { auth } from "@/lib/auth";

export async function requireUserId(): Promise<string> {
  const session = await auth.api.getSession({ headers: await headers() });
  if (!session) redirect("/login");
  return session.user.id;
}

8. 保護されたページの雛形

src/app/todos/page.tsx（Server Component）：

import { requireUserId } from "@/lib/session";

export default async function TodosPage() {
  const userId = await requireUserId();
  return <main className="p-8">ログイン中: {userId}</main>;
}

学びどころ：認可の基本形は「毎回サーバーで訊く」

クライアントの状態（「ログイン済みに見える」）は信用せず、ページと Server Action の入口で毎回 requireUserId() を呼ぶのが基本形です。 middleware でのガードは UX 向上のための補助であって、認可の本体はあくまで各エントリポイント側に置きます。

チェックポイント

  • サインアップ → Drizzle Studio の user テーブルに行が増える
  • ログイン → /todos に自分の userId が表示される
  • ブラウザのシークレットウィンドウで /todos を開く → /login にリダイレクトされる
  • ログアウト後に /todos → リダイレクトされる

つまずきポイント（山場その 1 の核心）

  • CLI の generate が DB 接続やスキーマ配置をうまく認識しないことがある。その場合は公式ドキュメントの Drizzle アダプタのページにある手書きスキーマ例をコピーして済ませてよい。CLI との格闘に時間を溶かさないこと。
  • auth.api.getSession に headers() を渡し忘れる／await headers() にしていない、が定番のエラー。Next.js 15 から headers() は async。
  • ID の型の不一致：Better Auth 生成テーブルの user.id は text。todos 側の userId も text にしてあるので合うはずだが、FK エラーが出たら型を最初に疑う。
