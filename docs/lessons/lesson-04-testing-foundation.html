Lesson 4: テスト基盤（Vitest × テスト用DB）

Vitest / Docker（テスト用 PostgreSQL）/ 統合テストのハーネス ｜ 目安 Day 8–11（約 4 時間）

ゴール

実 DB に対して走る統合テストの土台ができ、pnpm test がスモークテスト 1 本で緑になる状態。 以降の Lesson 5・6 で書く認可・upsert のテストは、すべてこの基盤の上に載せます。

学びどころ：このアプリのテストは「ダイヤモンド型」で積む

E2E（Playwright）
統合テスト（サービス関数 × 実 DB）
ユニット（純粋関数）

上下は薄く、中央（実 DB に対する統合）を最も太くする

ピラミッド（ユニットが最厚）ではなくダイヤモンドにするのは、このアプリで守りたい価値—— 認可（他人の todo に触れない）・タグ upsert（再利用と cascade）・トランザクション境界——が、 関数単体ではなく「実際の DB に対して SQL が正しく効いているか」でしか検証できないからです。 モックした DB でユニットテストを厚く書いても、この核心は守れません。

テスト対象は次の 2 種類に絞ります：

  • 純粋ヘルパー（normalizeTagNames・todoSchema の境界値）→ ユニット（下・薄）。
  • サービス関数（createTodoForUser など、next/* に依存しないロジック）→ 実 DB に対する統合（中央・太）。

学びどころ：Server Component はユニットテストしない（公式の推奨に従う）

Next.js 公式ドキュメントは、async Server Component はユニットテストより E2E を推奨すると明言しています—— 「Since async Server Components are new to the React ecosystem, some tools do not fully support them. In the meantime, we recommend using End-to-End Testing over Unit Testing for async components.」。 Vitest も現状 async Server Component を未サポートです。

本アプリの app/todos/page.tsx は「requireUserId() → db.query → 描画」という薄いガワにすぎず、 テストしたい認可・upsert・トランザクションは Server Action と DB ヘルパー側にあります。 だから Server Component は E2E（Lesson 10）だけで担保し、ロジックはサービス層に抽出して統合テストする——という切り分けが公式推奨と一致します。

Day 8: Vitest を入れて最小設定で緑にする

1. インストール

pnpm add -D vitest vite-tsconfig-paths dotenv-cli

vite-tsconfig-paths は @/* のパスエイリアスを Vitest でも解決するため。dotenv-cli は後述のテスト DB へのマイグレーションで使います。

2. vitest.config.ts

リポジトリ直下に作成します。環境は node（DOM は要らない。テスト対象はサーバー側のロジックと DB）：

import { config } from "dotenv";
import { defineConfig } from "vitest/config";
import tsconfigPaths from "vite-tsconfig-paths";

// テスト用の接続情報などを読み込む
config({ path: ".env.test" });

export default defineConfig({
  plugins: [tsconfigPaths()],
  test: {
    environment: "node",
    setupFiles: ["./src/test/setup.ts"],
    // 統合テストは同じ DB を共有するので、ファイル間を直列に走らせる
    fileParallelism: false,
  },
});

3. package.json にスクリプト

"scripts": {
  "test": "vitest"
}

この時点で pnpm test を叩くと「テストが無い」と言われます。DB の準備を整えてから最初の 1 本を書きます。

Day 9: テスト用 DB を Docker で開発用と分離する

なぜ DB を分けるのか

統合テストは beforeEach でテーブルを TRUNCATE します。もし開発用 DB に対して走らせたら、 手で入れた確認用データが毎回消えます。テスト用 DB は開発用と物理的に別ポート・別 DB にして、 「テストが触ってよいのはテスト DB だけ」を仕組みで保証します。

4. compose.yaml にテスト用サービスを足す

Lesson 2 の compose.yaml に db_test を追加します（別ポート 5433・別 DB 名）：

services:
  db:
    image: postgres:16
    environment:
      POSTGRES_USER: postgres
      POSTGRES_PASSWORD: postgres
      POSTGRES_DB: braindump_todo
    ports:
      - "5432:5432"
    volumes:
      - pgdata:/var/lib/postgresql/data
  db_test:
    image: postgres:16
    environment:
      POSTGRES_USER: postgres
      POSTGRES_PASSWORD: postgres
      POSTGRES_DB: braindump_todo_test
    ports:
      - "5433:5432"
    tmpfs:
      - /var/lib/postgresql/data   # テスト DB は揮発でよい（tmpfs で高速化・永続化しない）
volumes:
  pgdata:
docker compose up -d

5. .env.test に接続先を書く

リポジトリ直下に .env.test（.env と同様 gitignore 対象にする）：

TEST_DATABASE_URL=postgres://postgres:postgres@localhost:5433/braindump_todo_test

6. @/db を env で接続先切替できる形にする

Lesson 2 の src/db/index.ts を、テスト時だけ接続先を切り替えられるようにします。drizzle(pool, { schema }) の形はそのまま：

import { drizzle } from "drizzle-orm/node-postgres";
import { Pool } from "pg";
import * as schema from "./schema";

// テスト実行時は TEST_DATABASE_URL を最優先で使う。
// これが「開発 DB を誤って触らない」第一のガードになる。
const connectionString = process.env.TEST_DATABASE_URL ?? process.env.DATABASE_URL;
if (!connectionString) throw new Error("DATABASE_URL is not set");

const pool = new Pool({ connectionString });

export const db = drizzle(pool, { schema });

通常の pnpm dev では TEST_DATABASE_URL は未定義なので DATABASE_URL（開発用 Docker）を使い、 Vitest では .env.test が TEST_DATABASE_URL を与えるのでテスト DB に向きます。

Day 10: マイグレーション適用と truncate ハーネス

7. テスト DB にスキーマを適用する

テスト DB は生まれたての空なので、スキーマを流し込みます。テストでは履歴管理は不要なので、素早い push で十分です。 テスト用の drizzle 設定を 1 枚足します。drizzle.test.config.ts：

import { config } from "dotenv";
import { defineConfig } from "drizzle-kit";

config({ path: ".env.test" });

export default defineConfig({
  schema: "./src/db/schema.ts",
  out: "./drizzle",
  dialect: "postgresql",
  dbCredentials: { url: process.env.TEST_DATABASE_URL! },
});

package.json にスクリプトを足して実行します：

"db:push:test": "drizzle-kit push --config drizzle.test.config.ts"
pnpm db:push:test

スキーマ（Lesson 2・3 で定義した user / todos / tags / todos_to_tags など）が変わるたびに、 このコマンドでテスト DB を追従させます。

8. truncate ハーネス（テスト間の独立性）

src/test/setup.ts。全テストの前に共通で走る設定です：

import { beforeEach } from "vitest";
import { sql } from "drizzle-orm";
import { db } from "@/db";

// 第二のガード：TEST_DATABASE_URL が無ければ即座に落とす。
// 開発 DB を空にする事故を、TRUNCATE に到達する前に止める。
if (!process.env.TEST_DATABASE_URL) {
  throw new Error(
    "TEST_DATABASE_URL が未設定です。テストはテスト用 DB でのみ実行してください。",
  );
}

beforeEach(async () => {
  // "user" は SQL 予約語なので必ずクォートする。
  // RESTART IDENTITY で連番をリセット、CASCADE で依存テーブル（session 等）も一括で空にする。
  await db.execute(
    sql`TRUNCATE TABLE "user", todos, tags, todos_to_tags RESTART IDENTITY CASCADE`,
  );
});

学びどころ：なぜ「モック」ではなく「実 DB を truncate」なのか

DB をモックすると、検証したい肝心の部分——onConflictDoNothing がユニーク制約に本当に衝突するか、 onDelete: "cascade" が本当に中間テーブルを消すか、複合 WHERE が本当に他人の行を除外するか——が すべてモックの作り物になり、テストの意味が消えます。 実 DB に対して beforeEach で空にしながら走らせれば、「本物の PostgreSQL がどう振る舞うか」をそのまま検証できます。 これがダイヤモンドの中央を実 DB で太く積む理由です。

9. ユーザー seed ヘルパー

Lesson 3 で todos.userId / tags.userId は user.id への FK を張りました。 そのため todo やタグを入れる前に、必ず対応する user 行が要ります。以降のテストで使い回すヘルパーを用意します。src/test/seed.ts：

import { db } from "@/db";
import { user } from "@/db/schema";

// Better Auth の user テーブルに最小限の行を作り、その id を返す
export async function seedUser(email: string): Promise<string> {
  const id = crypto.randomUUID();
  await db.insert(user).values({
    id,
    name: email.split("@")[0],
    email,
    emailVerified: true,
    createdAt: new Date(),
    updatedAt: new Date(),
  });
  return id;
}

Day 11: 最初のスモーク統合テスト

10. ハーネスが DB まで通ることを証明する

src/db/schema.test.ts。1 テーブルへ insert → select し直して assert するだけの最小テストです：

import { describe, expect, it } from "vitest";
import { and, eq } from "drizzle-orm";
import { db } from "@/db";
import { tags } from "@/db/schema";
import { seedUser } from "@/test/seed";

describe("テストハーネスの疎通", () => {
  it("insert したタグを select し直せる", async () => {
    const userId = await seedUser("smoke@example.com");

    await db.insert(tags).values({ userId, name: "買い物" });

    const rows = await db
      .select()
      .from(tags)
      .where(and(eq(tags.userId, userId), eq(tags.name, "買い物")));

    expect(rows).toHaveLength(1);
    expect(rows[0].name).toBe("買い物");
  });
});
pnpm test

緑になれば、「Vitest → @/db → テスト DB → SQL 実行 → 結果を assert」の一本道が通ったことの証明です。 Lesson 5・6 はこの上に認可・upsert のテストを載せていきます。

チェックポイント

  • docker compose up -d で db（5432）と db_test（5433）の 2 つが立っている
  • pnpm db:push:test でテスト DB にテーブルが作られる（Drizzle Studio 等で 5433 を覗いて確認）
  • pnpm test がスモークテスト 1 本で緑になる
  • 開発用 DB（5432）に手で入れたデータが、テストを走らせても消えていない（＝別 DB になっている証拠）

つまずきポイント

  • 接続先ガードを外さない。setup.ts の TEST_DATABASE_URL チェックと、db/index.ts の切替。この 2 段のガードが「本番・開発 DB を誤って TRUNCATE する」最悪の事故を防ぐ。ローカルの小さなミスが CI で本番 DB を消す、が最も怖い。
  • vitest.config.ts の environment を node にする。既定の jsdom のままだと pg の接続まわりで不可解なエラーになる。
  • @/db が解決できない（Cannot find module）場合は vite-tsconfig-paths の plugin 登録忘れ。
  • スキーマを変えたのにテストが古いカラムで落ちるときは pnpm db:push:test の流し忘れ。テスト DB は自動追従しない。
  • ポート 5433 が既に使われていると db_test が起動しない。docker compose ps で状態を確認する。
