Lesson 6: AWS への初回デプロイ

SST / OpenNext / Aurora Serverless v2 ｜ 目安 Day 14–16（約 3 時間）＝ 既知の山場その 2

ゴール

ここまでのアプリ（認証＋CRUD＋タグ）が、CloudFront の URL で Aurora を DB として動いている状態。

先に知っておくコストの話

リソースコスト感
Lambda + CloudFront + S3低トラフィックならほぼ無料枠内
Aurora（0 ACU〜、auto-pause）停止中の計算コスト 0。ストレージ課金は継続（月 $1〜数ドル）
VPC の NATnat: "ec2"（t4g.nano）で月 $3〜4 程度。マネージド NAT Gateway（月 $30〜）は選ばない

遊び終わったら npx sst remove で全リソースを削除できます（これも IaC の学びの一部）。

Day 14: SST の初期化と構成

1. 初期化

npx sst@latest init

Next.js プロジェクトを自動検出し、sst.config.ts が生成されます。

2. sst.config.ts を書く

/// <reference path="./.sst/platform/config.d.ts" />

export default $config({
  app(input) {
    return {
      name: "braindump-todo",
      removal: input?.stage === "production" ? "retain" : "remove",
      home: "aws",
    };
  },
  async run() {
    const vpc = new sst.aws.Vpc("Vpc", { bastion: true, nat: "ec2" });

    const database = new sst.aws.Aurora("Database", {
      engine: "postgres",
      vpc,
      scaling: { min: "0 ACU", max: "4 ACU" },
    });

    const openaiApiKey = new sst.Secret("OpenaiApiKey");

    new sst.aws.Nextjs("Web", {
      vpc,
      link: [database, openaiApiKey],
    });
  },
});

学びどころ：SST v3 の 3 つの中心概念

  • コンポーネント — sst.aws.Aurora は「クラスタ＋サブネットグループ＋セキュリティグループ…」を 1 つにまとめた高水準部品。scaling.min: "0 ACU" で auto-pause が有効になる（requirements.md の決定事項どおり）。
  • link — リソース間の配線。link: [database] と書くだけで、接続情報が Lambda に注入され、かつ必要な IAM 権限とネットワーク設定が付与される。環境変数を手でコピペする世界からの脱却。
  • Secret — npx sst secret set OpenaiApiKey sk-... --stage dev で登録し、コードから Resource 経由で読む。secret はステージごとに登録する（--stage を省略すると personal ステージに入ってしまい、dev ステージのアプリからは読めない）。.env をデプロイ先に持ち込まない。

3. アプリ側から接続情報を読む

src/db/index.ts を、link された Aurora とローカル Docker の両対応にします：

import { drizzle } from "drizzle-orm/node-postgres";
import { Pool } from "pg";
import * as schema from "./schema";

function buildConnectionConfig() {
  // ローカル開発（Docker Postgres）は DATABASE_URL を使う
  if (process.env.DATABASE_URL) {
    return { connectionString: process.env.DATABASE_URL };
  }
  // SST 環境では link された Aurora から取る
  const { Resource } = require("sst") as typeof import("sst");
  return {
    host: Resource.Database.host,
    port: Resource.Database.port,
    user: Resource.Database.username,
    password: Resource.Database.password,
    database: Resource.Database.database,
  };
}

const pool = new Pool({ ...buildConnectionConfig(), max: 1 });

export const db = drizzle(pool, { schema });

学びどころ：Lambda では接続プールを小さく

max: 1 は Lambda の実行モデルに合わせた設定です。Lambda は 1 インスタンス＝同時 1 リクエストなので、 インスタンスごとに大きなプールを持つと、スケール時に Aurora の接続上限を食い潰します。 本格運用では RDS Proxy を挟みますが、トレーニング用途では max: 1 で十分。

Day 15: デプロイとマイグレーション

4. シークレットの登録とデプロイ

npx sst secret set OpenaiApiKey sk-xxxx --stage dev   # 値は Lesson 7 まで仮でも可
npx sst deploy --stage dev

初回は 15〜30 分かかります（Aurora クラスタと CloudFront の作成が遅い）。コーヒーを淹れながら、出力されるリソース名を眺めて「何が作られているか」を追ってください。完了すると CloudFront の URL が表示されます。

5. Aurora へのマイグレーション

Aurora は VPC 内にいるため、手元からは直接届きません。sst.config.ts の bastion: true がここで効きます：

npx sst tunnel --stage dev

トンネルが開いている間、link されたリソースへ手元から接続できます。別ターミナルで、Aurora 向けの接続情報を使って migrate を実行します（tunnel 起動時に表示される接続先を DATABASE_URL に組み立てて渡すのが手っ取り早い）：

DATABASE_URL="postgres://（tunnel が示す接続情報）" npx drizzle-kit migrate

学びどころ：マイグレーションファイル方式がここで報われる

Lesson 2 で push ではなく generate → migrate を選んだ理由がこれです。drizzle/ に積んだ SQL ファイルの列を そのまま Aurora に適用すれば、ローカルとまったく同じスキーマが再現されます。「DB が 2 つになった瞬間、 マイグレーションは履歴で管理するしかなくなる」ことを体感してください。

6. Better Auth の URL 設定

デプロイ先の URL を Better Auth に教える必要があります。sst.config.ts の Nextjs コンポーネントに環境変数を追加：

new sst.aws.Nextjs("Web", {
  vpc,
  link: [database, openaiApiKey],
  environment: {
    BETTER_AUTH_URL: "https://（CloudFront の URL）",
    BETTER_AUTH_SECRET: process.env.BETTER_AUTH_SECRET!,
  },
});

（process.env.BETTER_AUTH_SECRET は sst.config.ts の実行時に .env から読まれます——SST は .env を自動ロードします。きれいにやるなら secret を sst.Secret にもう 1 本切る。URL は初回デプロイ後に判明するので、2 回目のデプロイで設定する流れで OK）

Day 16: 動作確認と後片付けの練習

7. 本番相当での動作確認

CloudFront URL でサインアップ〜todo 作成〜タグ付けまで一通り実行します。

Aurora のコールドスタートを「観測」する

しばらく（デフォルトでは非アクティブ数分〜）放置すると Aurora は auto-pause します。 その状態で最初のリクエストを送ると十数秒待たされるはず——これは requirements.md で許容済みの想定挙動です。 ただし Server Action がタイムアウトする場合があるので、体感しておくこと。リロードで復帰します。 「初回だけ遅い」を実際に観測することが、このアーキテクチャを選んだ判断の意味を理解する近道です。

8. sst dev を知っておく（実行は任意）

sst dev は「インフラは AWS 上の実物、Next.js は手元で実行」というハイブリッドな開発モードで、link されたリソース（Aurora、Secret）に手元のコードから Resource でアクセスできます。ただし今回の構成では 2 つ注意があります：

  • ステージ指定なしの npx sst dev は personal ステージ（ユーザー名）に別スタックを新規作成します。VPC と Aurora クラスタがもう 1 組でき、課金も増えます。かといってデプロイ済みの dev ステージと sst dev を同一ステージで共用するのも非推奨です。
  • 手順 3 の buildConnectionConfig() は DATABASE_URL を優先するため、.env に DATABASE_URL がある限り、sst dev で起動しても接続先は Docker Postgres のままです。

したがって Lesson 7〜8 の開発は Docker Postgres 相手の npm run dev で進めます。sst dev を試すのは自由ですが、終わったら personal ステージのリソースを npx sst remove で必ず片付けてください。

チェックポイント

  • CloudFront URL でアプリ全機能が動く
  • Aurora が auto-pause → 再開する挙動を観測した
  • AWS コンソールの Billing で課金状況を一度確認した
  • sst.config.ts の各行が何を作っているか説明できる

つまずきポイント（山場その 2 の核心）

  • デプロイ失敗時はまずエラーメッセージのリソース名を見る。SST は Pulumi ベースなので、失敗しても再実行すれば途中から続きをやってくれる（冪等）。慌てて remove しない。
  • Lambda から DB に届かない（接続タイムアウト）場合は、Nextjs コンポーネントに vpc を渡し忘れているのが定番。
  • 認証だけ壊れる場合は BETTER_AUTH_URL が CloudFront の URL と一致しているか、Cookie の secure 属性まわりを疑う。
  • ハマったら 30 分で切り上げて SST の Discord／GitHub Discussions を検索する。初回デプロイの詰まりどころはほぼ既出。
