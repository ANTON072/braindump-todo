学びどころ：Conform の設計思想（先にこれを理解する）

Conform の核心は「同じ Zod スキーマで、クライアントとサーバーの両方で検証する」ことです。

  • クライアント側：useForm の onValidate で parseWithZod → 即時のエラー表示（UX）。
  • サーバー側：Server Action の先頭で同じスキーマの parseWithZod → 本物の防御（セキュリティ）。
  • 検証 NG のとき、サーバーは submission.reply() を返し、Conform がそれをフォームのエラー表示に復元する。この lastResult の往復が Conform のイディオムの中心。

Day 12: スキーマと作成フォーム

1. インストールと Zod スキーマ

pnpm add @conform-to/react @conform-to/zod zod@3

Zod は v4 でエラー指定が { error: ... } に変わり、Conform 側も import パスが @conform-to/zod/v4 になるなど、本テキストのコード（v3 構文）と互換がありません。そのため zod@3 を明示しています。

src/features/todos/schema.ts。クライアント・サーバーの両方から import するので、DB コードとは分離しておきます。

import { z } from "zod";

export const todoSchema = z.object({
  title: z.string({ required_error: "タイトルは必須です" }).max(200),
  notes: z.string().max(2000).optional(),
  dueDate: z.string().date().optional(),
  priority: z.enum(["low", "med", "high"]).default("med"),
});

2. Server Action（作成）

src/features/todos/actions.ts：

"use server";

import { revalidatePath } from "next/cache";
import { parseWithZod } from "@conform-to/zod";
import { db } from "@/db";
import { todos } from "@/db/schema";
import { requireUserId } from "@/lib/session";
import { todoSchema } from "./schema";

export async function createTodo(prevState: unknown, formData: FormData) {
  const userId = await requireUserId();

  const submission = parseWithZod(formData, { schema: todoSchema });
  if (submission.status !== "success") {
    return submission.reply();
  }

  await db.insert(todos).values({ ...submission.value, userId });
  revalidatePath("/todos");
  return submission.reply({ resetForm: true });
}

3. フォームコンポーネント

src/features/todos/todo-form.tsx（Client Component）：

"use client";

import { useActionState } from "react";
import { useForm, getFormProps, getInputProps, getSelectProps } from "@conform-to/react";
import { parseWithZod } from "@conform-to/zod";
import { createTodo } from "./actions";
import { todoSchema } from "./schema";
import { Button } from "@/components/ui/button";
import { Input } from "@/components/ui/input";

export function TodoForm() {
  const [lastResult, action] = useActionState(createTodo, undefined);
  const [form, fields] = useForm({
    lastResult,
    defaultValue: { priority: "med" },
    onValidate({ formData }) {
      return parseWithZod(formData, { schema: todoSchema });
    },
    shouldValidate: "onBlur",
  });

  return (
    <form {...getFormProps(form)} action={action} className="space-y-3">
      <Input {...getInputProps(fields.title, { type: "text" })} placeholder="やること" />
      <p className="text-sm text-red-600">{fields.title.errors}</p>
      <Input {...getInputProps(fields.dueDate, { type: "date" })} />
      <select {...getSelectProps(fields.priority)} className="border rounded p-2">
        <option value="low">低</option>
        <option value="med">中</option>
        <option value="high">高</option>
      </select>
      <Button type="submit">追加</Button>
    </form>
  );
}

学びどころ：getFormProps / getInputProps は何をしているか

id・name・aria-invalid・defaultValue 等の属性一式を、スキーマと検証状態から機械的に生成しています。 手書きの name="title" がスキーマとずれる事故（フォーム開発の定番バグ）を型レベルで防ぐのが狙い。 また getFormProps(form) に含まれる noValidate により、ブラウザ標準の検証を切って Zod に一本化しています。

Day 13: 一覧表示（Read）

4. 一覧ページ

src/app/todos/page.tsx。Server Component から直接 Drizzle を呼びます。

import { desc, eq } from "drizzle-orm";
import { db } from "@/db";
import { todos } from "@/db/schema";
import { requireUserId } from "@/lib/session";
import { TodoForm } from "@/features/todos/todo-form";
import { TodoItem } from "@/features/todos/todo-item";

export default async function TodosPage() {
  const userId = await requireUserId();
  const todoRows = await db.query.todos.findMany({
    where: eq(todos.userId, userId),
    orderBy: [desc(todos.createdAt)],
  });

  return (
    <main className="mx-auto max-w-2xl space-y-6 p-8">
      <TodoForm />
      <ul className="space-y-2">
        {todoRows.map((todo) => (
          <TodoItem key={todo.id} todo={todo} />
        ))}
      </ul>
    </main>
  );
}

学びどころ：一覧の認可は「WHERE 句」でやる

「全部取ってからフィルタ」ではなく、クエリの WHERE に必ず userId を入れる。 取得段階で他人のデータがプロセスに入ってこないようにするのが認可の第一原則です。

Day 14: 更新・削除をサービス層に抽出し、所有者チェックを寄せる

5. ロジックをサービス層に抽出する

完了切替・削除の本体を、next/* に一切依存しないサービス関数として src/features/todos/service.ts に切り出します。 DB ハンドル（db でも transaction の tx でも）・userId・対象 id を引数で受け取り、所有者チェックの複合 WHERE はここに置きます：

import { and, eq } from "drizzle-orm";
import { db } from "@/db";
import { todos } from "@/db/schema";

// db でも transaction でも受けられるように型を緩める
type Database = typeof db | Parameters<Parameters<typeof db.transaction>[0]>[0];

export async function toggleTodoStatusForUser(
  database: Database,
  userId: string,
  todoId: string,
) {
  const todo = await database.query.todos.findFirst({
    where: and(eq(todos.id, todoId), eq(todos.userId, userId)),
  });
  if (!todo) return { updated: false };

  await database
    .update(todos)
    .set({ status: todo.status === "open" ? "done" : "open", updatedAt: new Date() })
    .where(and(eq(todos.id, todoId), eq(todos.userId, userId)));
  return { updated: true };
}

export async function deleteTodoForUser(
  database: Database,
  userId: string,
  todoId: string,
) {
  const deleted = await database
    .delete(todos)
    .where(and(eq(todos.id, todoId), eq(todos.userId, userId)))
    .returning({ id: todos.id });
  return { deleted: deleted.length > 0 };
}

6. Server Action は薄いガワにする

actions.ts は「requireUserId → サービス呼び出し → revalidatePath」だけに保ちます。Conform を通さない単純操作なので、bind で id を渡す素朴な形で十分です：

"use server";

import { revalidatePath } from "next/cache";
import { db } from "@/db";
import { requireUserId } from "@/lib/session";
import { deleteTodoForUser, toggleTodoStatusForUser } from "./service";

export async function toggleTodoStatus(todoId: string) {
  const userId = await requireUserId();
  await toggleTodoStatusForUser(db, userId, todoId);
  revalidatePath("/todos");
}

export async function deleteTodo(todoId: string) {
  const userId = await requireUserId();
  await deleteTodoForUser(db, userId, todoId);
  revalidatePath("/todos");
}

学びどころ：所有者チェックの型 —「id だけで WHERE しない」（ロジックはサービス関数側に置く）

更新・削除の WHERE は常に and(eq(id, ...), eq(userId, ...)) の複合条件にします。 「先に SELECT して所有者を確かめてから UPDATE」でも間違いではありませんが、WHERE に含める形なら チェックと操作が単一のクエリで原子的になり、書き忘れも起きにくい。今回のアプリ全編を貫く型です。

この認可ロジックを Server Action ではなく service.ts に抜き出したのは、この分離が next/*（requireUserId・revalidatePath）をモックせずにテストできる理由でもあるからです。 サービス関数は DB ハンドルと userId を引数で受けるだけの純粋なロジックなので、Lesson 4 の実 DB ハーネスにそのまま渡して検証できます。 認可はこのアプリのセキュリティの核心——目視では抜け落ちるので、Day 15–16 で必ずテストで守ります。

Day 15–16: 認可を統合テストで守る（★セキュリティ最重要）

「他人の todo に触れない」は、目視確認では最も抜けやすく、しかし漏れると最悪の欠陥です。 Lesson 4 のハーネス（実 DB ＋ seedUser ＋ beforeEach の truncate）の上に、認可の統合テストを載せます。

8. 2 ユーザーを seed して境界を検証する

src/features/todos/service.test.ts。サービス関数は DB ハンドルと userId を引数で受けるだけなので、next/* を一切モックせずに実 DB で検証できます：

import { beforeEach, describe, expect, it } from "vitest";
import { and, eq } from "drizzle-orm";
import { db } from "@/db";
import { todos } from "@/db/schema";
import { seedUser } from "@/test/seed";
import { deleteTodoForUser, toggleTodoStatusForUser } from "./service";

let userA: string;
let userB: string;
let todoOfB: string;

beforeEach(async () => {
  userA = await seedUser("a@example.com");
  userB = await seedUser("b@example.com");
  const [todo] = await db
    .insert(todos)
    .values({ userId: userB, title: "B のタスク" })
    .returning();
  todoOfB = todo.id;
});

describe("削除の認可", () => {
  it("他人（A）は B の todo を削除できない", async () => {
    const result = await deleteTodoForUser(db, userA, todoOfB);

    expect(result.deleted).toBe(false);
    // B の todo は残っている
    const rows = await db.select().from(todos).where(eq(todos.id, todoOfB));
    expect(rows).toHaveLength(1);
  });

  it("所有者（B）自身は削除できる", async () => {
    const result = await deleteTodoForUser(db, userB, todoOfB);

    expect(result.deleted).toBe(true);
    const rows = await db.select().from(todos).where(eq(todos.id, todoOfB));
    expect(rows).toHaveLength(0);
  });
});

describe("完了切替の認可", () => {
  it("他人（A）は B の todo を切り替えられない", async () => {
    const result = await toggleTodoStatusForUser(db, userA, todoOfB);

    expect(result.updated).toBe(false);
    const [row] = await db.select().from(todos).where(eq(todos.id, todoOfB));
    expect(row.status).toBe("open"); // 変わっていない
  });
});

9. 一覧が WHERE userId で分離されることを検証する

一覧取得（Day 13）の where: eq(todos.userId, userId) が、他人の行を混ぜないことも実 DB で確かめます：

describe("一覧取得の分離", () => {
  it("自分の todo だけが返り、他人の分は混ざらない", async () => {
    await db.insert(todos).values({ userId: userA, title: "A のタスク" });

    const listForA = await db
      .select()
      .from(todos)
      .where(eq(todos.userId, userA));

    expect(listForA).toHaveLength(1);
    expect(listForA[0].title).toBe("A のタスク");
    // beforeEach で入れた B の todo は含まれない
    expect(listForA.every((t) => t.userId === userA)).toBe(true);
  });
});

学びどころ：認可テストは「他人になりすまして操作を試みる」形で書く

認可のテストの勘所は、成功ケースより「攻撃者（別ユーザー）が操作を試みて失敗する」ケースにあります。 deleteTodoForUser(db, userA, todoOfB) のように「A の資格で B の資源を触ろうとする」呼び出しが、 戻り値でも DB の実状態でも弾かれていることを両方 assert する。 複合 WHERE の eq(userId, ...) を誰かがうっかり消しても、このテストが赤くなって気づけます——これが回帰テストで認可を守るということです。

チェックポイント

  • 作成 → 一覧に反映され、フォームがリセットされる
  • タイトル空で送信 → クライアントで即エラー表示。さらに JS を無効化して送信してもサーバーで弾かれる（二重検証の確認）
  • ユーザーを 2 人作り、互いの todo が見えないこと・触れないことを確認
  • DevTools から他人の todoId で削除アクションを叩いても失敗する
  • pnpm test で認可・一覧分離の統合テストが緑になる（上の 2 項目を自動化したもの。今後の回帰はこのテストが守る）

つまずきポイント

  • Server Action の第 1 引数は prevState。useActionState と組む関数は (prevState, formData) のシグネチャにする。忘れると formData に prevState が入って混乱する。
  • submission.reply({ resetForm: true }) を返さないと、成功後もフォームに入力が残る。
  • revalidatePath を忘れると DB は更新されるのに画面が変わらない。「保存はされてるのに表示されない」ときはまずこれ。
