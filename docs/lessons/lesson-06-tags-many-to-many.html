Day 17: タグ upsert ヘルパー

1. 保存ロジックの設計を言葉で押さえる

「タグ名の配列を受け取り、todo に紐付ける」処理は 3 ステップに分解できます：

  1. INSERT ... ON CONFLICT DO NOTHING — 全タグ名を一括 INSERT。既存の (userId, name) は Lesson 2 で張ったユニーク制約に衝突して静かにスキップされる。
  2. SELECT で引き直す — 新規・既存を問わず、対象タグの id を名前でまとめて取得する。
  3. 中間テーブルへ INSERT — (todoId, tagId) を一括 INSERT。こちらも複合主キーとの衝突は DO NOTHING で流す。

学びどころ：なぜ「INSERT してから SELECT し直す」のか

onConflictDoNothing().returning() は実際に挿入された行しか返しません。既存タグの id は returning では手に入らない。 だから「新規も既存も区別せず、最後に名前で SELECT し直す」のがこのパターンの定石です。 1 件ずつ「SELECT → なければ INSERT」をループする素朴な実装と比べ、クエリ数が件数に依存しない（常に 3 回）のも利点。

2. 実装

src/features/tags/upsert-tags.ts。トランザクションの中から呼べるよう、第 1 引数で tx を受け取る形にします。

import { and, eq, inArray } from "drizzle-orm";
import { db } from "@/db";
import { tags, todosToTags } from "@/db/schema";

type Tx = Parameters<Parameters<typeof db.transaction>[0]>[0];

export function normalizeTagNames(raw: string): string[] {
  const names = raw
    .split(",")
    .map((name) => name.trim().toLowerCase())
    .filter((name) => name.length > 0);
  return [...new Set(names)];
}

export async function attachTagsToTodo(
  tx: Tx,
  userId: string,
  todoId: string,
  tagNames: string[],
) {
  if (tagNames.length === 0) return;

  // 1. あれば衝突スキップ、なければ作成
  await tx
    .insert(tags)
    .values(tagNames.map((name) => ({ userId, name })))
    .onConflictDoNothing();

  // 2. 新規・既存を問わず id を引き直す
  const tagRows = await tx
    .select({ id: tags.id })
    .from(tags)
    .where(and(eq(tags.userId, userId), inArray(tags.name, tagNames)));

  // 3. 中間テーブルに紐付け
  await tx
    .insert(todosToTags)
    .values(tagRows.map((tag) => ({ todoId, tagId: tag.id })))
    .onConflictDoNothing();
}

4. 作成ロジックを createTodoForUser に抽出する

Lesson 5 で作った src/features/todos/service.ts に、todo 本体の INSERT とタグ紐付けをひとまとめにする createTodoForUser を足します。 トランザクションの tx を受け取り、内部で attachTagsToTodo を呼ぶ形にします：

import { attachTagsToTodo, normalizeTagNames } from "@/features/tags/upsert-tags";
import { db } from "@/db";
import { todos } from "@/db/schema";

type Tx = Parameters<Parameters<typeof db.transaction>[0]>[0];

type CreateTodoInput = {
  title: string;
  dueDate?: string;
  priority?: "low" | "med" | "high";
  tagsInput?: string;
};

export async function createTodoForUser(
  tx: Tx,
  userId: string,
  input: CreateTodoInput,
) {
  const { tagsInput, ...todoValues } = input;
  const [todo] = await tx
    .insert(todos)
    .values({ ...todoValues, userId })
    .returning();
  await attachTagsToTodo(tx, userId, todo.id, normalizeTagNames(tagsInput ?? ""));
  return todo;
}

actions.ts の createTodo は、トランザクションを開いてこのサービス関数を呼ぶだけの薄いガワにします：

import { createTodoForUser } from "./service";

export async function createTodo(prevState: unknown, formData: FormData) {
  const userId = await requireUserId();

  const submission = parseWithZod(formData, { schema: todoSchema });
  if (submission.status !== "success") return submission.reply();

  await db.transaction((tx) => createTodoForUser(tx, userId, submission.value));

  revalidatePath("/todos");
  return submission.reply({ resetForm: true });
}

学びどころ：トランザクションの境界の決め方（と、それがテスト容易性に効く理由）

「todo は作られたのにタグの紐付けだけ失敗した」という中途半端な状態を許さないため、 todo 本体の INSERT とタグ処理を 1 つの db.transaction に入れます。 ヘルパーとサービス関数が db ではなく tx を受け取る設計にしたのはこのためで、「トランザクションを開く責任は呼び出し側、 中で何をするかは部品側」という分担は実務でもよく使う形です。

この分担はそのままテスト容易性にも効きます。createTodoForUser(tx, userId, input) は next/* に依存しないので、 テストからは db.transaction を開いて tx を渡すだけで、実 DB に対して作成＋タグ upsert の一連を検証できます（Day 20–21）。 Server Action ごとモックする必要はありません。

Day 19: タグ付きで一覧を取得する

5. リレーショナルクエリ

src/app/todos/page.tsx の取得部を差し替え：

const todoRows = await db.query.todos.findMany({
  where: eq(todos.userId, userId),
  orderBy: [desc(todos.createdAt)],
  with: {
    todosToTags: {
      with: { tag: true },
    },
  },
});

// 中間テーブル 2 段ネストを UI 用にフラット化する
const todosWithTags = todoRows.map((todo) => ({
  ...todo,
  tags: todo.todosToTags.map((link) => link.tag),
}));

表示は shadcn の Badge でタグ名を並べるだけで十分です。

学びどころ：with の 2 段ネストは Lesson 2 の relations の写し絵

with: { todosToTags: { with: { tag: true } } } という形は、relations() で宣言した 「todos → todosToTags → tag」の経路をそのままなぞっています。SQL 的には JOIN 相当の処理を Drizzle が組み立て、結果を型付きのネスト構造で返してくれる——これがリレーショナルクエリ API の価値です。 返り値の型にカーソルを乗せて、todosToTags: { tag: { name: string ... } }[] まで推論されていることを確認してください。

Day 20–21: upsert を統合テストで、正規化をユニットテストで守る

このレッスンの緑チェックポイント（再利用・cascade・userId 込みユニーク・空配列ガード）は、目視だと毎回手で確かめるのが大変で、しかも回帰しやすい箇所です。 そのままテストケースに写し取ります——中央（統合）を太く、下（ユニット）を薄く、というダイヤモンドの配分どおりに。

7. 統合テスト（中央・太）— 実 DB に対して upsert の性質を検証する

src/features/todos/create-todo.test.ts。createTodoForUser を db.transaction 越しに呼び、Lesson 4 のハーネスの実 DB で 4 つの性質を確かめます：

import { beforeEach, describe, expect, it } from "vitest";
import { eq } from "drizzle-orm";
import { db } from "@/db";
import { tags, todosToTags } from "@/db/schema";
import { seedUser } from "@/test/seed";
import { createTodoForUser, deleteTodoForUser } from "./service";

let userA: string;
let userB: string;

beforeEach(async () => {
  userA = await seedUser("a@example.com");
  userB = await seedUser("b@example.com");
});

describe("タグ upsert（createTodoForUser 経由）", () => {
  it("同名タグを含む 2 件を作っても tags は再利用され行が増えない", async () => {
    await db.transaction((tx) =>
      createTodoForUser(tx, userA, { title: "牛乳", tagsInput: "買い物" }));
    await db.transaction((tx) =>
      createTodoForUser(tx, userA, { title: "卵", tagsInput: "買い物, 仕事" }));

    const tagRows = await db.select().from(tags).where(eq(tags.userId, userA));
    // 「買い物」は再利用されるので、増えるのは「仕事」だけ → 計 2 行
    expect(tagRows).toHaveLength(2);
    expect(tagRows.map((t) => t.name).sort()).toEqual(["仕事", "買い物"]);
  });

  it("todo を削除すると中間テーブルは消えるが tags 本体は残る（cascade）", async () => {
    const todo = await db.transaction((tx) =>
      createTodoForUser(tx, userA, { title: "牛乳", tagsInput: "買い物" }));

    await deleteTodoForUser(db, userA, todo.id);

    const links = await db
      .select().from(todosToTags).where(eq(todosToTags.todoId, todo.id));
    expect(links).toHaveLength(0); // 紐付けは cascade で消える

    const tagRows = await db.select().from(tags).where(eq(tags.userId, userA));
    expect(tagRows).toHaveLength(1); // タグ本体は残る（他 todo で再利用できる）
  });

  it("別ユーザーは同名タグ「買い物」を独立して持てる（ユニークは userId 込み）", async () => {
    await db.transaction((tx) =>
      createTodoForUser(tx, userA, { title: "A の牛乳", tagsInput: "買い物" }));
    await db.transaction((tx) =>
      createTodoForUser(tx, userB, { title: "B の牛乳", tagsInput: "買い物" }));

    const allBuy = await db.select().from(tags).where(eq(tags.name, "買い物"));
    expect(allBuy).toHaveLength(2); // A 用・B 用で別行になる
  });

  it("タグ入力が空でもエラーにならず、todo だけ作られる（空配列ガード）", async () => {
    const todo = await db.transaction((tx) =>
      createTodoForUser(tx, userA, { title: "タグなし", tagsInput: "" }));

    expect(todo.title).toBe("タグなし");
    const tagRows = await db.select().from(tags).where(eq(tags.userId, userA));
    expect(tagRows).toHaveLength(0);
  });
});

学びどころ：この 4 本が「実 DB でしか守れない」性質そのもの

再利用は onConflictDoNothing ＋ 複合ユニーク制約が、cascade は onDelete: "cascade" が、 userId 込みユニークは uniqueIndex(userId, name) が、それぞれ PostgreSQL 側で効いて初めて成立します。 DB をモックしたら、これらは全部「モックがそう振る舞うと決めた作り物」になり、検証の意味が消えます。 だからここは中央（統合）で太く積む——ダイヤモンドの一番おいしいところです。

8. ユニットテスト（下・薄）— normalizeTagNames の正規化

正規化は DB を要さない純粋関数なので、こちらは軽いユニットで十分です。src/features/tags/upsert-tags.test.ts：

import { describe, expect, it } from "vitest";
import { normalizeTagNames } from "./upsert-tags";

describe("normalizeTagNames", () => {
  it("前後の空白を trim する", () => {
    expect(normalizeTagNames(" 買い物 , 仕事 ")).toEqual(["買い物", "仕事"]);
  });
  it("英字は小文字化する", () => {
    expect(normalizeTagNames("Work, HOME")).toEqual(["work", "home"]);
  });
  it("trim・小文字化した後に重複を除去する", () => {
    expect(normalizeTagNames("Work, work,  WORK ")).toEqual(["work"]);
  });
  it("空要素は落とす", () => {
    expect(normalizeTagNames("買い物, , ,仕事,")).toEqual(["買い物", "仕事"]);
  });
  it("空文字は空配列を返す（空配列ガードの前提）", () => {
    expect(normalizeTagNames("")).toEqual([]);
  });
});

この薄いユニットが、統合テストで守る「空配列ガード」の入口（正規化結果が空になりうる）を裏側で保証します。

チェックポイント

  • 「買い物, 仕事」で todo を 2 件作成 → tags テーブルの行は 2 行のまま増えない（再利用の確認）
  • 一覧の各 todo にタグの Badge が表示される
  • todo を削除 → todos_to_tags の紐付け行が消える（cascade の確認）が、tags 本体は残る
  • 別ユーザーで同名タグ「買い物」を作れる（ユニーク制約が userId 込みであることの確認）
  • pnpm test が緑になる。上の 4 項目は Day 20–21 の統合テスト 4 本にそのまま 1 対 1 対応している（手で確かめたことを、以後は回帰テストが守る）

つまずきポイント

  • inArray(tags.name, []) のように空配列を渡すとエラーになる。ヘルパー冒頭の length === 0 ガードを消さないこと。
  • タグ名の正規化（trim・小文字化・重複除去）を怠ると「買い物」と「買い物 」が別タグになる。正規化は保存直前の 1 箇所（normalizeTagNames）に集約する。
  • db.query が undefined になる場合は、Lesson 2 の drizzle(pool, { schema }) で schema を渡し忘れている。
