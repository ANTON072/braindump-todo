ゴール

todo が数千件あっても一覧が速い。クエリがインデックスに乗り、一覧はページネーションで必要な分だけ取る。 そして Next.js のキャッシュがどこで効き／効かないかを理解し、認証ページを誤ってキャッシュして他人のデータを出す事故を避けられる状態。

学びどころ：速さは「測ってから」。3 大地雷を知る

  • 無制限取得 — Lesson 5・6 の db.query.todos.findMany({ where: eq(todos.userId, userId) }) は件数の上限が無い。10 件なら快適でも、5000 件になった瞬間に一覧が固まる、典型的な「時限爆弾」です。
  • フルスキャン — インデックスが無いと、DB は毎回テーブル全走査＋ソートをする。
  • N+1 — 一覧の各行ごとに追加クエリ。今回はリレーショナルクエリ（with）で回避済みですが、油断すると復活します。

このレッスンは、今埋め込まれている1 つ目の地雷（無制限取得）を実際に処理し、2 つ目（インデックス）を EXPLAIN で可視化します。 「推測するな、計測せよ」を手を動かして体感します。

Day 46: カーソルページネーション

1. 一覧取得をサービス関数にして、上限とカーソルを入れる

一覧ロジックを src/features/todos/service.ts の listTodosForUser に集約します。 キーセット（カーソル）方式——「前ページの最後の行より後ろを N 件」——で取ります。同時刻の行がありうるので、createdAt と id の 2 段で並べ、カーソルも 2 段で比較します：

import { and, desc, eq, lt, or } from "drizzle-orm";
import { todos } from "@/db/schema";

const PAGE_SIZE = 20;

type Cursor = { createdAt: Date; id: string };

export async function listTodosForUser(
  database: Database,
  userId: string,
  cursor?: Cursor,
) {
  const rows = await database.query.todos.findMany({
    where: and(
      eq(todos.userId, userId),
      // カーソルより「後ろ」だけ。createdAt が同値なら id で決める
      cursor
        ? or(
            lt(todos.createdAt, cursor.createdAt),
            and(eq(todos.createdAt, cursor.createdAt), lt(todos.id, cursor.id)),
          )
        : undefined, // 初回はカーソル無し（and は undefined を無視する）
    ),
    orderBy: [desc(todos.createdAt), desc(todos.id)],
    limit: PAGE_SIZE + 1, // 「次があるか」を知るため 1 件多く取る
    with: { todosToTags: { with: { tag: true } } },
  });

  const hasMore = rows.length > PAGE_SIZE;
  const items = hasMore ? rows.slice(0, PAGE_SIZE) : rows;
  const last = items.at(-1);
  const nextCursor = hasMore && last
    ? { createdAt: last.createdAt, id: last.id }
    : null;

  return { items, nextCursor };
}

学びどころ：なぜ OFFSET ではなくカーソルか

LIMIT 20 OFFSET 4000 は、DB が4000 件を読み飛ばしてから20 件返すため、後ろのページほど遅くなります。 さらに、読んでいる間に新しい行が入るとページがズレて重複・抜けが起きます。 カーソル方式は「最後に見た位置より後ろ」を WHERE で直接指すので、何ページ目でも速度が一定で、挿入があってもズレません。 無限スクロールや「もっと見る」と相性がよいのはこのためです。

2. ページと UI をつなぐ

app/todos/page.tsx は searchParams からカーソルを受け取り、listTodosForUser を呼ぶだけにします。 「もっと見る」は nextCursor を ?cursor= に載せたリンク、または useActionState で追記読み込みにします（実装はどちらでも可）。 カーソルは createdAt と id の 2 値なので、encodeURIComponent(JSON.stringify(cursor)) のように 1 パラメータにまとめると扱いやすいです。

Day 47: インデックス設計

3. 一覧クエリに効くインデックスを張る

一覧は「userId で等価 → createdAt, id で降順ソート」です。この形に一致する複合インデックスを schema.ts に足します：

import { index } from "drizzle-orm/pg-core";

export const todos = pgTable(
  "todos",
  {
    // ...（既存のカラム定義はそのまま）
  },
  (t) => [
    index("todos_user_created_idx").on(t.userId, t.createdAt.desc(), t.id.desc()),
  ],
);

pnpm exec drizzle-kit migrate

4. EXPLAIN で「効いた」ことを目で確認する

インデックスの前後で実行計画を比べます（Drizzle Studio か psql で）：

EXPLAIN ANALYZE
SELECT * FROM todos
WHERE user_id = '（自分の userId）'
ORDER BY created_at DESC, id DESC
LIMIT 21;

インデックス前は Seq Scan＋Sort、張った後は Index Scan になり、Sort が消えるはずです（並び順がインデックスと一致するため）。

学びどころ：複合インデックスの列順は「等価 → ソート」

複合インデックスは左から順に使われます。今回は userId（等価で絞る）を先頭に、続いて並べ替えに使う createdAt DESC, id DESC を置く。 この順序だと「userId で範囲を特定 → その中は既に並んでいる」ので、DB は追加のソートをせず先頭 21 件を返せます。 Lesson 2 の tags(userId, name) ユニークインデックスは制約（重複を防ぐ）が目的でしたが、こちらは性能が目的——同じ「インデックス」でも狙いが違うことを区別してください。

Day 48: Next.js のキャッシュモデル

学びどころ：4 つのキャッシュと、認証アプリでの効き方

キャッシュ層今回のアプリでは
Request Memoization1 リクエスト内同じ関数の重複呼び出しをまとめる。常に有効
Data Cacheサーバ・永続fetch 単位。今回は DB 直アクセスなので基本使わない
Full Route Cacheサーバ・ビルド/再検証時効かない。認証で headers()/cookies() を読むため動的レンダリングになる
Router Cacheクライアント遷移時の一時キャッシュ。revalidatePath で無効化される

本アプリの /todos は requireUserId() がリクエストの Cookie を読むため、Next.js が自動で動的レンダリングと判断し、 ページ全体はキャッシュされません。だから毎回「その人の」データが出ます。 Lesson 5 以降で使ってきた revalidatePath("/todos") の正体は、クライアントの Router Cache（と該当パスのデータ）を無効化して、次の表示で最新を取り直させる指示だったわけです。

5. 認証ページを「静的化しない」ことを保証する

誤キャッシュ＝他人のデータ流出になりうる

認証付きページを何かの拍子に静的化・キャッシュしてしまうと、ある人向けに作った HTML が別の人に配られる——認可（Lesson 5）を丁寧に作っても、キャッシュ 1 つで台無しです。 今回は Cookie を読むので自動で動的になりますが、明示のガードとして /todos のページに次を置いておくと事故を防げます：

// app/todos/page.tsx
export const dynamic = "force-dynamic";

逆に、ログイン不要の完全公開ページ（ランディング等）があれば、そこは静的化・キャッシュの恩恵を積極的に受けてよい。 「ユーザー固有か・公開か」でキャッシュ方針を分けるのが判断軸です。unstable_cache や fetch のキャッシュは、今回のように「ユーザー固有で可変」なデータには使いません。

Day 49: 計測と仕上げ

6. 大量データで体感し、ページネーションをテストで守る

Lesson 4 のハーネスに、境界を跨ぐページングの統合テストを足します。createdAt が同時刻に固まっても id で正しく分割できること（重複・欠落なし）を確かめます。src/features/todos/list.test.ts：

import { beforeEach, describe, expect, it } from "vitest";
import { db } from "@/db";
import { todos } from "@/db/schema";
import { seedUser } from "@/test/seed";
import { listTodosForUser } from "./service";

let userId: string;

beforeEach(async () => {
  userId = await seedUser("perf@example.com");
  // 25 件（PAGE_SIZE=20 を跨ぐ。createdAt は defaultNow で近接する）
  for (let i = 0; i < 25; i++) {
    await db.insert(todos).values({ userId, title: `todo ${i}` });
  }
});

describe("カーソルページネーション", () => {
  it("2 ページで全 25 件を重複・欠落なく取得できる", async () => {
    const page1 = await listTodosForUser(db, userId);
    expect(page1.items).toHaveLength(20);
    expect(page1.nextCursor).not.toBeNull();

    const page2 = await listTodosForUser(db, userId, page1.nextCursor!);
    expect(page2.items).toHaveLength(5);
    expect(page2.nextCursor).toBeNull(); // 最終ページ

    const ids = [...page1.items, ...page2.items].map((t) => t.id);
    expect(new Set(ids).size).toBe(25); // 重複なし・全件
  });
});

手元でさらに数千件を seed して EXPLAIN ANALYZE の実測時間を見ると、インデックスとページネーションの効果が数字で分かります。

チェックポイント

  • 一覧が PAGE_SIZE 件で頭打ちになり、「もっと見る」で続きが取れる
  • EXPLAIN ANALYZE がインデックス後に Index Scan になり、Sort が消える
  • ページネーションの境界テストが緑（重複・欠落なし）
  • /todos が動的レンダリングで、別ユーザーのデータがキャッシュから出ないことを確認
  • 数千件 seed しても一覧の初期表示が速いままである

つまずきポイント

  • カーソルの並び（createdAt, id）と orderBy、そしてインデックスの列順が三者一致していないと、ソートが消えず速くならない。
  • createdAt 単独をカーソルにすると、同時刻の行が飛ぶ／重複する。必ず一意な id をタイブレークに含める。
  • 複合インデックスの列順を間違える（ソート列を先頭に置く等）と、等価の絞り込みに使われず効かない。
  • 認証ページに export const revalidate や fetch(..., { cache: "force-cache" }) をうっかり足すと静的化し、他人のデータが出る危険。ユーザー固有ページはキャッシュしない。
  • revalidatePath を忘れると、更新後に画面が変わらない（Lesson 5 既出）。これはキャッシュ無効化の問題だったと、ここで腑に落ちるはず。
