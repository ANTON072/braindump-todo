Lesson 13: 本番の可観測性（CloudWatch）

構造化ログ / CloudWatch Logs Insights / メトリクスフィルタ / アラート（SNS メール）｜ 目安 Day 42–45（約 4 時間）

ゴール

本番で「エラーが出た」「遅い」を能動的に知れる状態。ログは検索可能な構造化 JSON で相関 ID を持ち、 エラーが一定数を超えるとメールで通知され、届いた通知から CloudWatch Logs Insights で原因ログまで辿れる。 追加の外部サービスは使わず、AWS（SST）の中だけで完結させます。

学びどころ：可観測性の 3 本柱を「最小で」揃える

柱問い今回の実装
Logs「何が起きたか」構造化 JSON ログ → CloudWatch Logs（Lambda は stdout が自動収集される）
Metrics「どのくらいの頻度・速さか」ログから作るエラー件数メトリクス＋ Lambda / Aurora の標準メトリクス
Alerts「今すぐ見るべきか」閾値超過で SNS → メール通知

「デプロイできた（Lesson 7・11）」の次は「壊れたら気づける」。ここが趣味と実務の境目です。 Lambda + Aurora はログが分散しがちな構成なので、相関 IDで 1 リクエストの足跡を追えるようにするのが要になります。

Day 42: 構造化ログを出す

1. JSON 1 行で出すロガー

Lambda は stdout がそのまま CloudWatch Logs に流れます。1 行 1 JSONで出せば、後段の Logs Insights が各フィールドで検索・集計できます。src/lib/logger.ts：

type Level = "info" | "warn" | "error";

export function log(
  level: Level,
  message: string,
  meta: Record<string, unknown> = {},
) {
  const line = JSON.stringify({
    level,
    message,
    time: new Date().toISOString(),
    ...meta,
  });
  (level === "error" ? console.error : console.log)(line);
}

2. 相関 ID で 1 リクエストを束ねる

Server Action の入口で requestId を 1 つ発行し、その処理内のログすべてに付けます。actions.ts の例：

import { log } from "@/lib/logger";

export async function createTodo(prevState: unknown, formData: FormData) {
  const requestId = crypto.randomUUID();
  const userId = await requireUserId();
  log("info", "createTodo start", { requestId, userId });

  try {
    const submission = parseWithZod(formData, { schema: todoSchema });
    if (submission.status !== "success") return submission.reply();
    await db.transaction((tx) => createTodoForUser(tx, userId, submission.value));
    revalidatePath("/todos");
    log("info", "createTodo ok", { requestId, userId });
    return submission.reply({ resetForm: true });
  } catch (error) {
    // 詳細はここに集約。ユーザーには Lesson 12 の error.tsx / 文言で返す
    log("error", "createTodo failed", { requestId, userId, error: String(error) });
    throw error;
  }
}

学びどころ：ログに個人情報・秘密を書かない

ログは長期に残り、広く閲覧されます。パスワード・API キー・トークン・メール本文・生の個人情報は出さない。 今回 userId（不透明な ID）は出しますが、メールアドレスやタグの中身のような内容は原則出しません。 「相関に必要な最小限（id と種別）」だけを構造化して出す、が基本姿勢です。ログ量はそのままコストにもなります。

Day 43: CloudWatch Logs Insights で追う

3. エラーを一覧する

デプロイ後、AWS コンソールの CloudWatch → Logs Insights でサーバ関数のロググループを選び、JSON フィールドで検索します：

fields @timestamp, level, message, requestId, userId
| filter level = "error"
| sort @timestamp desc
| limit 50

4. 相関 ID で 1 リクエストを再構成する

エラー行から requestId を拾い、その ID の足跡だけを時系列で並べれば、「start → どこで failed したか」が 1 本の物語として読めます：

fields @timestamp, level, message
| filter requestId = "ここに拾った UUID"
| sort @timestamp asc

Lesson 12 で触れたとおり、本番でユーザーに投げた例外は Next.js が digest（相関 ID）だけ表に出します。 ユーザーから「この digest が出た」と言われたら、それをログの相関 ID と突き合わせて原因に辿り着ける——これが「見える化」の実利です。

Day 44: メトリクスとアラート

5. エラーログ件数をメトリクスにする

「エラーが出た」を待つのではなく、閾値超過で自分に通知させます。sst.config.ts の run() に、SNS トピック・メール購読・ログのメトリクスフィルタ・アラームを足します（SST v3 では aws プロバイダがグローバルに使えます）：

// 既存: vpc / database / openaiApiKey ...
const web = new sst.aws.Nextjs("Web", { vpc, link: [database, openaiApiKey] });

// 通知先（デプロイ後に届く確認メールで購読を承認する）
const alerts = new aws.sns.Topic("Alerts");
new aws.sns.TopicSubscription("AlertsEmail", {
  topic: alerts.arn,
  protocol: "email",
  endpoint: "you@example.com",
});

// サーバ関数のロググループに「level=error の JSON 行」を数えるフィルタを張る
const logGroupName = web.nodes.server.nodes.logGroup.name;
new aws.cloudwatch.LogMetricFilter("ErrorLogFilter", {
  logGroupName,
  pattern: '{ $.level = "error" }',
  metricTransformation: {
    name: "AppErrorCount",
    namespace: "BraindumpTodo",
    value: "1",
    defaultValue: "0",
  },
});

// 5 分で 5 件を超えたらメール
new aws.cloudwatch.MetricAlarm("ErrorRateAlarm", {
  namespace: "BraindumpTodo",
  metricName: "AppErrorCount",
  statistic: "Sum",
  period: 300,
  evaluationPeriods: 1,
  threshold: 5,
  comparisonOperator: "GreaterThanThreshold",
  treatMissingData: "notBreaching",
  alarmActions: [alerts.arn],
});

6. 標準メトリクスにもアラームを張る

アプリのログだけでなく、AWS が自動で出すメトリクスも監視対象です。最低限、次の 2 つを押さえます：

  • Lambda（AWS/Lambda）— Errors・Duration・Throttles。関数が例外で落ちた回数・遅さ・スロットリング。
  • Aurora（AWS/RDS）— ServerlessDatabaseCapacity（ACU）・DatabaseConnections。Lesson 7 で観測した auto-pause / 復帰や、接続の枯渇をここで数値で捉える。

同じ aws.cloudwatch.MetricAlarm の型で namespace・metricName を差し替えて足します。

Day 45: 運用をつなぐ（ランブックとダッシュボード）

7. ランブックを 1 枚書く

アラートは「来てから考える」と遅い。来たら何を見るかを短く決めておきます。docs/runbook.md（例）：

# アラート: ErrorRateAlarm（5分で5件超）
1. Logs Insights で level=error を直近 30 分で一覧（Day 43 のクエリ）
2. 最頻出の message と requestId を 1 つ選ぶ
3. requestId で足跡を時系列に並べ、失敗箇所を特定
4. 一時要因（Aurora コールドスタート等）なら Duration / ACU も確認
5. 修正 → PR → CI 緑 → staging 確認 → production 昇格（Lesson 11）

8. ダッシュボードでひと目にする（任意）

CloudWatch Dashboard に「エラー件数・Lambda Duration・Aurora ACU」を並べておくと、 アラートが鳴る前の傾向（じわじわ遅くなっている等）に気づけます。まずはコンソールで手作りで十分です。

学びどころ：この 3 レッスンが 1 本の輪になる

Lesson 12（壊れ方を優雅にする）でユーザー向けの安全な文言とdigestを出し、 Lesson 13（見える化）で開発者向けの詳細ログとアラートを用意し、 Lesson 11（CI/CD）で修正を安全に本番へ戻す。「壊れる → 気づく → 直す → 安全に出す」がひと回りして初めて、 運用できるアプリになります。作れるだけでは半分です。

チェックポイント

  • 本番のログが JSON 1 行で出ており、Logs Insights で level・requestId で絞り込める
  • わざとエラーを 5 分で 6 回起こす → アラートメールが届く
  • 届いた通知から Logs Insights → 相関 ID で原因ログまで 3 分以内に辿り着ける
  • ログにパスワード・API キー・メール本文などの機微情報が出ていない
  • Lambda Errors / Aurora ACU のアラームが張られている

つまずきポイント

  • SNS メール購読は確認メールのリンクを踏むまで有効化されない。「アラームは発火するのにメールが来ない」はまずこれ。
  • SST でサーバ関数のロググループ名を参照するノードのパス（本文の web.nodes.server.nodes.logGroup.name）は SST のバージョンで変わりうる。動かないときは sst のドキュメントや web.nodes の内容を確認して合わせる。
  • メトリクスフィルタの pattern は JSON 用の記法（{ $.level = "error" }）。ログが JSON でないとマッチしない＝ Day 42 の構造化が前提。
  • ログの出しすぎはコスト。ループ内で 1 件ずつ出す等は避け、相関に要る粒度に絞る。
  • ローカル（pnpm dev）では CloudWatch は無いので、ログはターミナルに出るだけ。可観測性の確認は staging / production で行う。
