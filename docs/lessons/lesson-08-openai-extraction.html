Day 25: 抽出スキーマと API 呼び出し

1. インストールと環境変数

pnpm add openai

.env に追記します（ローカル開発用。デプロイ環境では Lesson 7 で登録した Secret が使われます）：

OPENAI_API_KEY=sk-（自分のキー）

2. 抽出スキーマを Zod で定義する

src/features/braindump/schema.ts：

import { z } from "zod";

export const extractedTaskSchema = z.object({
  title: z.string().min(1),
  // Structured Outputs は optional を許さないため nullable で表現する（下の学びどころ参照）
  dueDate: z.string().nullable(),
  priority: z.enum(["low", "med", "high"]).nullable(),
  tags: z.array(z.string()),
});

export const extractionResultSchema = z.object({
  tasks: z.array(extractedTaskSchema),
});

export type ExtractedTask = z.infer<typeof extractedTaskSchema>;

学びどころ：Structured Outputs のスキーマ制約

OpenAI の Structured Outputs は JSON Schema の strict モードで動くため、独特の制約があります：

  • 全フィールド必須 — .optional() は使えない。「値がないかもしれない」は .nullable()（null を返させる）で表現する。
  • ルートはオブジェクト — 配列を直接返せないので { tasks: [...] } と 1 枚ラップする。

この「API の都合が混ざったスキーマ」と「アプリ内部のきれいな型」を混同しないことが、構造化出力を使う設計の勘所です。

3. 抽出関数

src/features/braindump/extract-tasks.ts：

import OpenAI from "openai";
import { zodTextFormat } from "openai/helpers/zod";
import { extractionResultSchema, type ExtractedTask } from "./schema";

function resolveOpenaiApiKey(): string {
  // ローカル開発（pnpm dev）は .env の値を使う
  if (process.env.OPENAI_API_KEY) return process.env.OPENAI_API_KEY;
  // SST 環境では link された Secret から取る（Lesson 7 の db/index.ts と同じ分岐）
  const { Resource } = require("sst") as typeof import("sst");
  return Resource.OpenaiApiKey.value;
}

const client = new OpenAI({ apiKey: resolveOpenaiApiKey() });

const SYSTEM_PROMPT = `あなたはタスク抽出アシスタントです。
ユーザーが雑に書き出したテキストから、実行可能なタスクを抽出してください。
- title は簡潔な命令形にする
- 期日が読み取れる場合のみ dueDate に YYYY-MM-DD で入れる（今日は {today}）
- 緊急度が読み取れる場合のみ priority を設定する
- 内容を分類する日本語タグを 0〜3 個付ける（例: 買い物, 仕事, 家庭）
- タスクでないもの（感想・メモ）は含めない`;

export async function extractTasksFromBraindump(
  rawText: string,
): Promise<ExtractedTask[]> {
  const today = new Date().toISOString().slice(0, 10);

  const response = await client.responses.parse({
    model: "gpt-4o-mini",
    input: [
      { role: "system", content: SYSTEM_PROMPT.replace("{today}", today) },
      { role: "user", content: rawText },
    ],
    text: { format: zodTextFormat(extractionResultSchema, "extraction") },
  });

  // SDK がパース済みでも、境界防御として自前の Zod で再検証する
  const verified = extractionResultSchema.safeParse(response.output_parsed);
  if (!verified.success) {
    throw new Error("抽出結果がスキーマに一致しませんでした");
  }
  return verified.data.tasks;
}

学びどころ：なぜ「再検証」するのか

Structured Outputs はスキーマ準拠を強く保証しますが、それでも LLM の出力は外部入力として扱うのが原則です。 SDK のパース結果を safeParse にもう一度通すことで、(1) SDK やモデルの挙動変更から下流コードを守る、 (2) 「この関数から出てくる値は必ずこの型」という保証を自分のコードベースの責任範囲で完結させる、の 2 つが得られます。 requirements.md が Zod を「OpenAI の構造化出力を再検証する層」と位置づけているのはこのためです。

Day 26: Server Action 化と実験

4. Server Action にする

src/features/braindump/actions.ts：

"use server";

import { requireUserId } from "@/lib/session";
import { extractTasksFromBraindump } from "./extract-tasks";

export async function runBraindumpExtraction(rawText: string) {
  await requireUserId();  // API コストを守るため、認証必須にする
  if (rawText.trim().length === 0 || rawText.length > 4000) {
    throw new Error("テキストは 1〜4000 文字で入力してください");
  }
  return extractTasksFromBraindump(rawText);
}

5. プロンプトで遊ぶ（ここが一番おもしろい）

UI がまだないので、仮ページかテストスクリプトから呼び出して実験します。試すと学びが多い入力：

  • 基本形：「歯医者に電話、牛乳買う、金曜までにレポート、Sam にメール」
  • 相対日付：「明日ゴミ出し、来週月曜に請求書」→ dueDate が正しく解決されるか（system prompt に today を渡した効果）
  • タスクでないもの混じり：「今日は疲れた。あと牛乳買う」→ 感想が除外されるか
  • 空っぽの入力：「あああ」→ tasks が空配列で返るか

期待とずれたら SYSTEM_PROMPT を調整します。スキーマ（構造の保証）とプロンプト（内容の質）は別のレイヤーで、前者は Zod が、後者はプロンプト調整が担当——という分担が体感できるはずです。

チェックポイント

  • 例文がタスク配列に構造化され、型が ExtractedTask[] で通る
  • 「金曜までに」等の相対日付が正しい YYYY-MM-DD になる
  • 未ログイン状態から Server Action を叩くとリダイレクト（コスト防御の確認）

つまずきポイント

  • Zod スキーマに .optional() や .default() が混ざっていると API がスキーマ変換エラーを返す。エラーメッセージに required と出たらこれ。
  • API キーは絶対にクライアントに置かない。呼び出しは必ず Server Action / サーバーコード内で。NEXT_PUBLIC_ プレフィックスを付けたら事故。
  • SDK のメソッド名（responses.parse / zodTextFormat）は SDK のバージョンで変わりやすい。動かない場合は openai-node の README の Structured Outputs 節と突き合わせる。
