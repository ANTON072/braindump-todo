Lesson 11: CI/CD と環境昇格

GitHub Actions / SST の stage（dev・staging・production）/ 承認ゲート ｜ 目安 Day 33–37（約 5 時間）

ゴール

Pull Request を出すと lint・型チェック・テスト（ユニット／統合／E2E）が自動で回り、緑でなければマージできない。 main にマージすると staging に自動デプロイされ、承認を挟んで production に昇格する。 環境は dev（各自の手元）／ staging ／ production に分かれ、secret も環境ごとに管理される状態。

学びどころ：CI と CD は別物。パイプラインは「速い門 → 遅い門 → デプロイ」

PR→ lint / typecheck / unit・統合→ E2E→ merge→ deploy staging→ 承認→ deploy production
  • CI（継続的インテグレーション）＝ 変更のたびに自動検証する門。速いもの（lint・型・ユニット・統合）を先に、遅くて重い E2E を後に置き、落ちたら先へ進めない。
  • CD（継続的デリバリー）＝ 緑になった変更を環境へ運ぶ。staging へは自動、production へは人の承認を挟む。

Lesson 4〜6・10 で書いたテストは、こうして自動で回して初めて意味が閉じます。手元で一度緑にして終わりでは、明日の自分や他人の変更を守れません。

Day 33–34: CI — PR で検証を自動化する

1. スクリプトを揃える

package.json に、CI から呼ぶ入口を用意します（lint は Lesson 1 の next lint、型チェックを 1 本足す）：

"scripts": {
  "lint": "next lint",
  "typecheck": "tsc --noEmit",
  "test": "vitest run",
  "db:push:test": "drizzle-kit push --config drizzle.test.config.ts"
}

CI では vitest run（1 回実行して終了）にします。ウォッチモードの vitest のままだと CI がハングします。

2. 速い門と統合テストのワークフロー

.github/workflows/ci.yml。統合テストは実 DB が要るので、GitHub Actions の service コンテナで PostgreSQL を立てます（Lesson 4 のテスト DB を CI 上に再現）：

name: CI
on:
  pull_request:

jobs:
  quality:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: pnpm/action-setup@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: pnpm
      - run: pnpm install --frozen-lockfile
      - run: pnpm lint
      - run: pnpm typecheck

  test:
    runs-on: ubuntu-latest
    services:
      postgres:
        image: postgres:16
        env:
          POSTGRES_USER: postgres
          POSTGRES_PASSWORD: postgres
          POSTGRES_DB: braindump_todo_test
        ports:
          - 5433:5432
        # DB が受付可能になるまで待つ（これが無いと接続拒否で不安定になる）
        options: >-
          --health-cmd pg_isready
          --health-interval 10s
          --health-timeout 5s
          --health-retries 5
    env:
      TEST_DATABASE_URL: postgres://postgres:postgres@localhost:5433/braindump_todo_test
    steps:
      - uses: actions/checkout@v4
      - uses: pnpm/action-setup@v4
      - uses: actions/setup-node@v4
        with: { node-version: 20, cache: pnpm }
      - run: pnpm install --frozen-lockfile
      - run: pnpm db:push:test    # テスト DB にスキーマを流す（Lesson 4）
      - run: pnpm test

3. E2E は重いので別ジョブに切る

同じ ci.yml にジョブを追加します。Playwright はブラウザの導入が要り、dev サーバも起動するので、速い門と分離して並行実行します：

  e2e:
    runs-on: ubuntu-latest
    services:
      postgres:
        image: postgres:16
        env:
          POSTGRES_USER: postgres
          POSTGRES_PASSWORD: postgres
          POSTGRES_DB: braindump_todo_test
        ports: [ "5433:5432" ]
        options: >-
          --health-cmd pg_isready --health-interval 10s
          --health-timeout 5s --health-retries 5
    env:
      CI: "true"                 # playwright.config の reuseExistingServer:false を効かせる
      TEST_DATABASE_URL: postgres://postgres:postgres@localhost:5433/braindump_todo_test
      # dev サーバ起動に必要な最小 env（本物である必要はない）
      BETTER_AUTH_SECRET: ci-dummy-secret-please-change
      BETTER_AUTH_URL: http://localhost:3000
      BETTER_AUTH_DISABLE_SIGNUP: "false"   # E2E の seed を許可（Lesson 10）
    steps:
      - uses: actions/checkout@v4
      - uses: pnpm/action-setup@v4
      - uses: actions/setup-node@v4
        with: { node-version: 20, cache: pnpm }
      - run: pnpm install --frozen-lockfile
      - run: pnpm exec playwright install --with-deps chromium
      - run: pnpm db:push:test
      - run: pnpm exec playwright test

学びどころ：ローカルの「テスト DB を分ける」設計が、そのまま CI に効く

Lesson 4 で @/db を TEST_DATABASE_URL 優先にし、E2E の dev サーバも同じ変数で切り替えられるようにしておいたおかげで、 CI では その環境変数を service コンテナに向けるだけで同じテストが動きます。 「どこで走っても同じ」を作るのがハーネス設計の目的で、CI はその最初のご褒美です。

4. main を保護する

GitHub のリポジトリ設定 → Branch protection rules で main に「Require status checks to pass（quality・test・e2e）」を必須化します。これで赤い PR は物理的にマージできません。

Day 35: 環境（stage）を分ける

学びどころ：SST の stage ＝ 環境

Lesson 7 では --stage dev だけを使いました。SST では stage 名がそのまま独立した環境（別 VPC・別 Aurora・別 secret）になります。今回は 3 つに分けます：

stage用途特徴
個人名（例 you）各自の実験sst dev や使い捨て。removal: "remove"
staging本番前の共有確認環境main マージで自動デプロイ
production本番承認後のみ。removal: "retain" で誤削除を防ぐ

5. sst.config.ts を環境差分に対応させる

Lesson 7 の app() は既に removal を stage で分けていました。ここに環境ごとの URL 差分などを足します。BETTER_AUTH_URL は環境で変わるので、run() の中で stage を見て組み立てます：

async run() {
  const vpc = new sst.aws.Vpc("Vpc", { bastion: true, nat: "ec2" });
  const database = new sst.aws.Aurora("Database", {
    engine: "postgres", vpc, scaling: { min: "0 ACU", max: "4 ACU" },
  });
  const openaiApiKey = new sst.Secret("OpenaiApiKey");

  const web = new sst.aws.Nextjs("Web", {
    vpc,
    link: [database, openaiApiKey],
    environment: {
      // URL は初回デプロイ後に確定するので、2 回目で本値に差し替える（Lesson 7 と同じ流儀）
      BETTER_AUTH_SECRET: new sst.Secret("BetterAuthSecret").value,
    },
  });

  return { url: web.url };
}

6. secret は stage ごとに登録する

Lesson 7 で触れたとおり、secret は stage ごとに別物です。staging・production の分を登録します：

pnpm dlx sst secret set OpenaiApiKey sk-xxxx --stage staging
pnpm dlx sst secret set BetterAuthSecret "$(openssl rand -base64 32)" --stage staging
pnpm dlx sst secret set OpenaiApiKey sk-yyyy --stage production
pnpm dlx sst secret set BetterAuthSecret "$(openssl rand -base64 32)" --stage production

Day 36–37: CD — 自動デプロイと承認ゲート

7. AWS 認証は OIDC で（長期キーを CI に置かない）

アクセスキーを GitHub Secrets に貼らない

長期の AWS_ACCESS_KEY_ID を CI に置くのは漏洩時の被害が大きい。GitHub Actions の OIDC を使い、 実行のたびに短命の一時クレデンシャルを AWS から受け取る形にします。AWS 側に GitHub を信頼する IAM ロール（gha-deploy など）を 1 つ作り、その ARN だけを CI に渡します。

8. デプロイのワークフロー

.github/workflows/deploy.yml。main マージで staging に出し、production は GitHub Environments の承認を通してから昇格します：

name: Deploy
on:
  push:
    branches: [ main ]

permissions:
  id-token: write   # OIDC で一時クレデンシャルを受け取るために必須
  contents: read

jobs:
  deploy-staging:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: pnpm/action-setup@v4
      - uses: actions/setup-node@v4
        with: { node-version: 20, cache: pnpm }
      - run: pnpm install --frozen-lockfile
      - uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: arn:aws:iam::<ACCOUNT_ID>:role/gha-deploy
          aws-region: ap-northeast-1
      - run: pnpm dlx sst deploy --stage staging

  deploy-production:
    needs: deploy-staging          # staging が成功しないと動かない
    runs-on: ubuntu-latest
    environment: production        # ← この環境に「必須レビュアー」を設定して承認ゲートにする
    steps:
      - uses: actions/checkout@v4
      - uses: pnpm/action-setup@v4
      - uses: actions/setup-node@v4
        with: { node-version: 20, cache: pnpm }
      - run: pnpm install --frozen-lockfile
      - uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: arn:aws:iam::<ACCOUNT_ID>:role/gha-deploy
          aws-region: ap-northeast-1
      - run: pnpm dlx sst deploy --stage production

学びどころ：承認ゲートは「人が最後に立つ場所」

GitHub の Environments で production に「Required reviewers」を設定すると、deploy-production ジョブは承認されるまで待機します。 自動化はしても、本番反映の最終判断だけは人が押す——この 1 クッションが、真夜中の事故を防ぎます。 なお、production へ出すのはテストが緑の mainだけ（CI が門番）で、staging での目視確認を経ている、という多重の防御になっています。

9. マイグレーションの扱い

スキーマ変更を含むデプロイでは、Lesson 7 の sst tunnel 経由の drizzle-kit migrate を各環境に対して実行します。 CI に組み込むこともできますが、DB 破壊のリスクが最も高い操作なので、本トレーニングでは migrate は手動のままにし、「アプリのデプロイ」と「スキーマ適用」を切り分けて考える癖をつけます。

チェックポイント

  • わざとテストを 1 本落とした PR を出す → CI が赤くなり、マージがブロックされる
  • PR を緑にして main にマージ → staging へ自動デプロイされ、staging の URL でアプリが動く
  • production ジョブが「承認待ち」で止まり、承認すると production に反映される
  • staging と production で別々の secret（OpenAI キー等）が使われている
  • CI に AWS の長期アクセスキーが 1 つも置かれていない（OIDC のみ）

つまずきポイント

  • service コンテナの Postgres に health check を付け忘れると、DB 起動前に接続して不定期に落ちる。--health-cmd pg_isready は必須。
  • vitest（ウォッチ）のまま CI に置くとジョブが終わらない。CI は必ず vitest run。
  • OIDC ロールの信頼ポリシーで、repo:<org>/<repo>:* の条件を絞らないと他リポジトリから assume される。ブランチや環境で条件を絞る。
  • Playwright のブラウザは --with-deps を付けないと Linux の共有ライブラリ不足で起動しない。
  • SST は同一 stage への同時デプロイでロックする。deploy ワークフローの並行実行は concurrency で 1 本に絞る。
