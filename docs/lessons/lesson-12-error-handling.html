Lesson 12: エラーハンドリングと失敗時の UX

App Router の error.tsx / loading.tsx / not-found.tsx / Server Action のエラー設計 / 外部依存の再試行 ｜ 目安 Day 38–41（約 4 時間）

ゴール

例外が起きてもユーザーに白画面やスタックトレースを見せず、意味のある画面（読み込み中・エラー・見つからない）が出る。 Server Action の失敗は握って UI に反映され、Aurora のコールドスタートや OpenAI の一時失敗は再試行で吸収される状態。

学びどころ：失敗を 3 つに分類し、「投げる」か「値で返す」かを決める

種類例扱い
予期した業務エラーバリデーション NG、権限なしで他人の todo を操作値で返す（例外にしない）。UI が分岐して表示する
見つからない存在しない todo の詳細ページnotFound() を呼ぶ → not-found.tsx
予期せぬ例外DB 接続断、想定外の null投げる → error.tsx が受け止める

この切り分けが全ての土台です。「なんでも throw new Error」でも「なんでも握りつぶす」でもなく、 ユーザーが次に何をすればいいかで扱いを決めます。Lesson 5 で deleteTodoForUser が例外ではなく { deleted: false } を返す設計にしたのは、まさに「予期した業務エラーは値で返す」だったわけです。

Day 38: ルートセグメントの UX 部品を置く

1. loading.tsx — 読み込み中の骨組み

Lesson 5 の app/todos/page.tsx は async Server Component で、DB クエリの間ユーザーを待たせます。同階層に loading.tsx を置くだけで、その待ち時間に Suspense フォールバックが自動で表示されます。app/todos/loading.tsx：

export default function Loading() {
  return (
    <main className="mx-auto max-w-2xl space-y-2 p-8">
      <div className="h-10 animate-pulse rounded bg-gray-100" />
      <div className="h-8 animate-pulse rounded bg-gray-100" />
      <div className="h-8 animate-pulse rounded bg-gray-100" />
    </main>
  );
}

2. error.tsx — 想定外の例外を受け止める

error.tsx は必ず Client Componentで、error と reset（再試行）を受け取ります。app/todos/error.tsx：

"use client";

import { useEffect } from "react";
import { Button } from "@/components/ui/button";

export default function Error({
  error,
  reset,
}: {
  error: Error & { digest?: string };
  reset: () => void;
}) {
  useEffect(() => {
    // 詳細はここでログ/監視へ送る（本番の中身は Lesson 13 で実装）
    console.error(error);
  }, [error]);

  return (
    <main className="mx-auto max-w-2xl space-y-4 p-8">
      <h2 className="text-lg font-bold">問題が発生しました</h2>
      <p className="text-sm text-gray-600">
        一時的な不具合の可能性があります。時間をおいて再度お試しください。
      </p>
      <Button onClick={() => reset()}>再試行</Button>
    </main>
  );
}

3. not-found.tsx と global-error.tsx

「見つからない」専用の画面と、ルートレイアウト自体が壊れた最後の砦を置きます。app/not-found.tsx：

import Link from "next/link";

export default function NotFound() {
  return (
    <main className="mx-auto max-w-2xl space-y-3 p-8">
      <h2 className="text-lg font-bold">ページが見つかりません</h2>
      <Link href="/todos" className="text-blue-600 underline">Todos に戻る</Link>
    </main>
  );
}

app/global-error.tsx はルートレイアウトの描画すら失敗した場合に出るため、自前で <html>・<body> を含めます（通常のレイアウトが使えないので）。

学びどころ：これらは「置くだけ」で App Router が配線する

loading / error / not-found はファイル名が規約で、そのセグメント配下を自動でラップします。 error.tsx は React の Error Boundary を、loading.tsx は Suspense 境界を Next.js が裏で張ってくれる仕組みです。 だから個別の try/catch を各所に撒く前に、まず境界ファイルを置くのが App Router の作法になります。

Day 39: Server Action の失敗を UI に返す

4. 完了切替・削除の結果を受け取る

Lesson 5 の toggleTodoStatus / deleteTodo は、サービス関数の戻り値（{ updated } / { deleted }）を捨てて revalidatePath するだけでした。 ここを useActionState で結果を受け取れる形にして、失敗をユーザーに見せます。actions.ts：

type ActionResult = { ok: true } | { ok: false; message: string };

export async function deleteTodo(
  prevState: unknown,
  todoId: string,
): Promise<ActionResult> {
  const userId = await requireUserId();
  const { deleted } = await deleteTodoForUser(db, userId, todoId);
  if (!deleted) {
    // 予期した業務エラー（他人の todo・既に消えている）は値で返す
    return { ok: false, message: "削除できませんでした" };
  }
  revalidatePath("/todos");
  return { ok: true };
}

TodoItem（Client Component）側は useActionState で受けて、失敗時だけメッセージを出します：

"use client";
import { useActionState } from "react";
import { deleteTodo } from "./actions";

export function DeleteButton({ todoId }: { todoId: string }) {
  const [state, action] = useActionState(deleteTodo, { ok: true } as const);
  return (
    <form action={action.bind(null, todoId)}>
      <button type="submit" className="text-sm text-gray-500">削除</button>
      {!state.ok && <p className="text-xs text-red-600">{state.message}</p>}
    </form>
  );
}

5. 「見つからない」は notFound() で表す

todo 詳細ページのように「その id が無ければ 404」の場面では、例外ではなく notFound() を使います：

import { notFound } from "next/navigation";

const todo = await db.query.todos.findFirst({
  where: and(eq(todos.id, id), eq(todos.userId, userId)),
});
if (!todo) notFound();   // → not-found.tsx が描画される

ここでも eq(todos.userId, userId) を効かせるのが肝心です。他人の todo は「権限なし」ではなく「存在しない（404）」として扱う——資源の存在自体を漏らさない、認可の定石です。

redirect() / notFound() を try/catch で握らない

Next.js の redirect()・notFound() は内部的に特殊な例外を投げて制御を移します。 これらの呼び出しを try/catch で囲むと、リダイレクトや 404 が「握りつぶされたエラー」になって動かなくなります。 Server Action で try { ... redirect() } catch と書きがちなので注意——redirect() は必ず try の外に置く。

Day 40: 外部依存の失敗に備える（コールドスタート・OpenAI）

6. Aurora のコールドスタートを再試行で吸収する

Lesson 7 で「auto-pause 明けの最初のリクエストは十数秒待たされる」と観測しました。ここでは握って再試行します。src/db/with-retry.ts：

// 一時的な接続エラー/タイムアウトだけを対象に、短い待機を挟んで数回だけ再試行する
export async function withRetry<T>(
  operation: () => Promise<T>,
  { retries = 2, delayMs = 1500 } = {},
): Promise<T> {
  let lastError: unknown;
  for (let attempt = 0; attempt <= retries; attempt++) {
    try {
      return await operation();
    } catch (error) {
      lastError = error;
      if (!isTransient(error) || attempt === retries) throw error;
      await new Promise((resolve) => setTimeout(resolve, delayMs));
    }
  }
  throw lastError;
}

function isTransient(error: unknown): boolean {
  const code = (error as { code?: string })?.code;
  // 接続まわりの一時エラーのみ（論理エラーは即失敗させる）
  return code === "ECONNREFUSED" || code === "ETIMEDOUT" || code === "57P01";
}

一覧取得のような読み取りを withRetry(() => db.query.todos.findMany(...)) で包みます。 再試行してよいのは冪等な操作（主に読み取り）だけ。副作用のある書き込みを無条件で再試行すると二重登録を招くので、書き込みはトランザクションと業務キーで守る前提にします。

7. OpenAI 呼び出しにタイムアウトと再試行を付ける

Lesson 8 の extractTasksFromBraindump は失敗すると素の例外でした。タイムアウトと1 回の再試行を足し、ユーザー向けには落ち着いた文言にします：

const client = new OpenAI({ apiKey: resolveOpenaiApiKey(), timeout: 20_000, maxRetries: 1 });

// runBraindumpExtraction 側（Server Action）
try {
  return { ok: true, tasks: await extractTasksFromBraindump(rawText) };
} catch (error) {
  console.error("braindump extraction failed", error); // 詳細はログへ（Lesson 13）
  return { ok: false, message: "抽出に失敗しました。少し時間をおいて再度お試しください。" };
}

学びどころ：本番の文言に内部情報を出さない

ユーザーに見せるのは「何をすればいいか」だけ（再試行・時間をおく）。スタックトレースや DB エラー文はユーザーに出さず、ログ／監視へ回すのが鉄則です。 Next.js も本番では Server Action / Server Component で投げた例外のメッセージを伏せ、digest（相関 ID）だけを表に出します。 「ユーザー向けの安全な文言」と「開発者向けの詳細」を分けて出す——その詳細の出し先を作るのが次の Lesson 13 です。

Day 41: 通しで壊して確かめる

実際に壊して、各境界が期待どおり出るかを確認します。壊し方の例：

  • 存在しない id で todo 詳細にアクセス → not-found.tsx が出る
  • deleteTodoForUser に他人の id を渡す（DevTools で改ざん）→ フォーム直下に「削除できませんでした」
  • Docker の DB を一時停止して一覧を開く → withRetry の待機の後、復帰するか error.tsx に落ちる
  • OpenAI キーをわざと不正にして抽出 → 落ち着いた文言が出て、アプリは壊れない

チェックポイント

  • 読み込み中に loading.tsx の骨組みが一瞬見える
  • 想定外の例外で白画面ではなく error.tsx（再試行ボタン付き）が出る
  • 存在しない／他人の資源は 404 として扱われ、存在の有無が漏れない
  • Server Action の失敗がフォームに反映され、成功時はリセット/更新される
  • DB コールドスタートや OpenAI 一時失敗が、再試行または丁寧な文言で吸収される

つまずきポイント

  • error.tsx に "use client" を付け忘れると動かない。Error Boundary はクライアント側の仕組み。
  • error.tsx は同階層の layout.tsx の中で描画されるため、レイアウト自体の例外は捕まえられない。それを捕まえるのが global-error.tsx（<html>/<body> 必須）。
  • 書き込み操作を安易に withRetry で包むと二重実行の恐れ。再試行は読み取り中心に。
  • 予期した業務エラーまで throw すると、毎回 error.tsx に飛んでしまい UX が荒くなる。「値で返す／投げる」の分類（冒頭の表）に立ち返る。
