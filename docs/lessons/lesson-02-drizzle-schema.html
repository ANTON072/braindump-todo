Lesson 2: DB スキーマとマイグレーション

Drizzle ORM / PostgreSQL (Docker) ｜ 目安 Day 3–4（約 2 時間）

ゴール

ローカルの Docker PostgreSQL に対して、todos / tags / todos_to_tags のテーブルがマイグレーションで作成され、Drizzle Studio で中身を覗ける状態。

方針：Lesson 5 まではローカルの Docker PostgreSQL で開発します。Aurora は Lesson 6 で登場します。 DB の向き先を接続文字列 1 本（DATABASE_URL）に寄せておくことで、後から Aurora に差し替えやすくします。

Day 3: 環境と設定

1. PostgreSQL を Docker で立てる

リポジトリ直下に compose.yaml を作成：

services:
  db:
    image: postgres:16
    environment:
      POSTGRES_USER: postgres
      POSTGRES_PASSWORD: postgres
      POSTGRES_DB: braindump_todo
    ports:
      - "5432:5432"
    volumes:
      - pgdata:/var/lib/postgresql/data
volumes:
  pgdata:
docker compose up -d

2. Drizzle のインストールと設定

pnpm add drizzle-orm pg
pnpm add -D drizzle-kit @types/pg

.env（gitignore 済みであることを確認）：

DATABASE_URL=postgres://postgres:postgres@localhost:5432/braindump_todo

リポジトリ直下に drizzle.config.ts：

import { defineConfig } from "drizzle-kit";

export default defineConfig({
  schema: "./src/db/schema.ts",
  out: "./drizzle",
  dialect: "postgresql",
  dbCredentials: { url: process.env.DATABASE_URL! },
});

3. DB クライアント

src/db/index.ts：

import { drizzle } from "drizzle-orm/node-postgres";
import { Pool } from "pg";
import * as schema from "./schema";

const pool = new Pool({ connectionString: process.env.DATABASE_URL });

export const db = drizzle(pool, { schema });

{ schema } を渡すのがポイント。これで後述のリレーショナルクエリ（db.query.todos.findMany）が型付きで使えるようになります。

Day 4: スキーマ定義とマイグレーション

4. スキーマを書く

src/db/schema.ts。users テーブルは Lesson 3 で Better Auth が生成するため、ここでは userId を text 型のプレーンなカラムとして先に定義し、FK は Lesson 3 で張ります。

import {
  pgTable, pgEnum, text, date, timestamp, primaryKey, uniqueIndex,
} from "drizzle-orm/pg-core";
import { relations } from "drizzle-orm";

export const priorityEnum = pgEnum("priority", ["low", "med", "high"]);
export const statusEnum = pgEnum("status", ["open", "done"]);

export const todos = pgTable("todos", {
  id: text("id").primaryKey().$defaultFn(() => crypto.randomUUID()),
  userId: text("user_id").notNull(),
  title: text("title").notNull(),
  notes: text("notes"),
  dueDate: date("due_date"),
  priority: priorityEnum("priority").notNull().default("med"),
  status: statusEnum("status").notNull().default("open"),
  createdAt: timestamp("created_at").notNull().defaultNow(),
  updatedAt: timestamp("updated_at").notNull().defaultNow(),
});

export const tags = pgTable(
  "tags",
  {
    id: text("id").primaryKey().$defaultFn(() => crypto.randomUUID()),
    userId: text("user_id").notNull(),
    name: text("name").notNull(),
  },
  (t) => [uniqueIndex("tags_user_id_name_unique").on(t.userId, t.name)],
);

export const todosToTags = pgTable(
  "todos_to_tags",
  {
    todoId: text("todo_id").notNull()
      .references(() => todos.id, { onDelete: "cascade" }),
    tagId: text("tag_id").notNull()
      .references(() => tags.id, { onDelete: "cascade" }),
  },
  (t) => [primaryKey({ columns: [t.todoId, t.tagId] })],
);

学びどころ：多対多の 3 点セット

  • 中間テーブル＋複合主キー — primaryKey({ columns: [todoId, tagId] }) により「同じ todo に同じタグを二重に付ける」ことが DB レベルで不可能になる。アプリ側の防御より強い保証。
  • 複合ユニーク制約 — tags(userId, name) のユニークインデックスが Lesson 5 の upsert（onConflictDoNothing）の衝突判定の土台になる。upsert は「ユニーク制約があって初めて成立する」と覚える。
  • onDelete: "cascade" — todo を消せば紐付け行も自動で消える。中間テーブルの掃除をアプリコードでやらずに済む。

5. リレーションを宣言する

同じ schema.ts に追記。これは FK とは別物で、リレーショナルクエリ API に「どう JOIN できるか」を教えるための宣言です。

export const todosRelations = relations(todos, ({ many }) => ({
  todosToTags: many(todosToTags),
}));

export const tagsRelations = relations(tags, ({ many }) => ({
  todosToTags: many(todosToTags),
}));

export const todosToTagsRelations = relations(todosToTags, ({ one }) => ({
  todo: one(todos, {
    fields: [todosToTags.todoId],
    references: [todos.id],
  }),
  tag: one(tags, {
    fields: [todosToTags.tagId],
    references: [tags.id],
  }),
}));

学びどころ：relations() は多対多では「1 対多を 2 回」書く

Drizzle には「many-to-many」を一発で書く構文はなく、todos → todosToTags（1対多）と todosToTags → tags（多対1）を組み合わせます。 そのため取得時のネストも with: { todosToTags: { with: { tag: true } } } と 2 段になります（Lesson 5 で実践）。

6. マイグレーションの生成と適用

pnpm exec drizzle-kit generate   # drizzle/ に SQL ファイルが生成される
pnpm exec drizzle-kit migrate    # DB に適用

生成された SQL を必ず開いて読むこと。CONSTRAINT ... PRIMARY KEY("todo_id","tag_id") のように、TypeScript の定義がどんな DDL に変換されたかを確認するのが一番の教材です。

7. Drizzle Studio で確認

pnpm exec drizzle-kit studio

ブラウザでテーブル一覧が見えます。手で 1〜2 行入れてみて、複合ユニーク制約に反する行（同じ userId＋name のタグ）が弾かれることも試してみてください。

チェックポイント

  • drizzle/ にマイグレーション SQL があり、内容を読んで説明できる
  • Studio で 3 テーブルが見える
  • tags に同一 (userId, name) を 2 回入れるとエラーになる

つまずきポイント

  • drizzle-kit generate と push の違い：push はマイグレーションファイルを作らず直接 DB を書き換える。試行錯誤には楽だが、本番（Aurora）にも同じ変更を再現する必要があるため、今回は generate → migrate の王道フローで統一する。
  • drizzle.config.ts が .env を読まない場合は pnpm add -D dotenv して config 先頭で import "dotenv/config"。
