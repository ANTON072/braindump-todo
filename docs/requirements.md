## 1. これは何のためのプロジェクトか（最重要）

これは **プロダクトを作ること自体が目的ではなく、以下のスタックを習得するためのトレーニング** です。
作者は 9 個のツールの多くを初めて触ります。したがって次の方針が全体を貫きます。

- **アプリ本体は意図的に小さく保つ。** 節約した時間をツールの習得に回す。
- **凝った実装より、各ツールの「王道・イディオム的な使い方」を優先する。** 後で読み返して学べることが価値。
- 実装を進めるときは、特に学びどころ（多対多の upsert、Conform の field array、SST の Aurora コンポーネント）で **何をしているか簡潔に説明** してほしい。黙って全部やってしまわない。

### 時間予算
- **1 日あたり約 1 時間 × 20 日 ＝ 実働およそ 20 時間** で「動くもの」を完成させたい。
- ロジック自体は小さいが、**時間の大半はツールの配線**（SST 初回デプロイ、Drizzle × Better Auth 接続、マイグレーション）に消える見込み。そこを織り込んで計画すること。

## 2. プロダクト概要

**Todo アプリ ＋ ブレインダンプ抽出 ＋ タグ（多対多）。**

- 見た目・操作感は「ふつうの Todo アプリ」。
- 目玉機能は **ブレインダンプ**：雑に書いた文章を貼り付けると、OpenAI が構造化タスクの配列に整形して返す。ユーザーが確認・修正してから保存する。
  - 例：「歯医者に電話、牛乳買う、金曜までにレポート、Sam にメール」→ `[{title, dueDate?, priority?, tags[]}, ...]` に構造化。
- **今回 OpenAI API は「構造化（抽出）」の用途で使う。** ブレインダンプで自由文をタスクの配列に整形するのが主目的。チャットボットや文章生成のような対話・生成系の機能は、今回の題材ではスコープ外とする。

## 4. インフラの決定事項（AWS・低コスト前提）

- デプロイ先は **OpenNext + SST** → Lambda + CloudFront + S3。低トラフィックなら計算コストはほぼ無料枠。
- **DB は SST v3 の Aurora コンポーネント（Aurora Serverless v2 / PostgreSQL）を使う。**
  - **最小 0 ACU / 最大 4 ACU、アイドル時に auto-pause**（SST の Aurora コンポーネントのデフォルトに一致）。
  - 停止中は **計算コストが 0**。ただし **ストレージ課金は継続**（小規模なら月 $1〜数ドル程度）。
  - **再開時に十数秒のコールドスタート**がある。トレーニング用途では許容。ただし「最初のリクエストが遅い」挙動は想定しておくこと。
- DB は **PostgreSQL**。今回はベクトル検索は使わない（タグ方式を採用したため pgvector 不要）。
- 環境は最低限で良い。まず **dev/personal ステージ 1 つを通す**ことを優先。

## 5. データモデル（提案・出発点）

> これは会話で合意した方向性を反映した **たたき台**。カラムの過不足は Claude Code がプランニング時に精査してよい。

- **users** — Better Auth が管理（Better Auth が自前のテーブル群を作る。Drizzle アダプタで接続）。
- **todos**
  - `id`, `userId`(FK), `title`, `notes?`, `dueDate?`, `priority`(enum: low/med/high), `status`(enum: 例 open/done), `createdAt`, `updatedAt`
- **tags**
  - `id`, `userId`(FK), `name`
  - **ユーザーごとにユニーク**（`userId` + `name` の複合ユニーク）
- **todos_to_tags**（中間テーブル）
  - `todoId`(FK), `tagId`(FK)、**複合主キー**

### 学習上の主眼（ここが今回いちばん身になる箇所）
- 多対多を **中間テーブル＋複合主キー** で設計する。
- 取得は **Drizzle のリレーショナルクエリ**（`db.query.todos.findMany({ with: { tags: true } })` 的な書き方）で「todo にタグをぶら下げて取る」。
- ブレインダンプ保存時に、抽出された各タグ名について **「あれば取得・なければ作成（find-or-create / upsert）→ 中間テーブルで紐付け」**。実務頻出のパターンで、ここが最重要の学びどころ。**時間が押しやすいのもこの保存ロジック。**

### 認可
- **自分のデータだけ見える／触れる。** Server Action 内で todo・tag の所有者チェックを行う。

## 6. ブレインダンプのパイプライン（目玉機能）

```
貼り付けたテキスト
  → OpenAI（Structured Outputs / JSON schema で配列を返させる）
  → Zod で再検証
  → Conform の field array に「編集可能なリスト」としてプリフィル
  → ユーザーが確認・修正
  → Server Action で保存（tags は upsert して中間テーブルに紐付け）
```

- 抽出スキーマ：`{ title: string, dueDate?: string, priority?: enum, tags: string[] }` の配列。
- **請求書の明細行のような相互依存（金額計算など）はない** ので、Conform の field array を「やさしい形」で学べる。

## 7. スコープ優先度（遅延時の切り分け）

計画・実装は必ずこの優先度に沿って、**上から順に「動く状態」を積み上げる**こと。下位が未完でも上位だけで成立する状態を保つ。

1. **【必達コア】** 認証（Better Auth: メール＋パスワード）＋ todos の CRUD（Conform + Zod + Server Actions + Drizzle）＋ 自分のデータのみの認可。
2. **【必達コア】** tags（多対多）：スキーマ・保存（upsert＋紐付け）・タグ付きでの一覧取得。
3. **【目玉・できたら】** ブレインダンプ抽出（OpenAI → Zod → Conform field array → 保存）。
4. **【おまけ・任意】** タグでの絞り込み UI、凝ったタグ入力（チップ型ピッカー）など **UI の作り込み**。

> **DB 側（多対多の設計・保存・取得）は学習価値が高いのでしっかりやる。**
> **タグの入力 UI や絞り込みは時間が漏れやすいので最小限**（素朴な複数選択やカンマ区切りで可）。ここは 4 番＝任意。

## 8. 非ゴール（作らないもの）

- チャットボット・文章生成などの対話・生成系の機能。
- リスト/プロジェクトによるグループ化（1 対多）。※タグがグループ化の役割も兼ねるため、今回は入れない。余力があれば将来。
- チーム・組織・共有（マルチテナント）。今回は単一ユーザーが自分のデータを扱うだけ。
- 通知、リマインダー、繰り返しタスク等の付加機能。
